## 解决的问题

ACP/daemon 路径仍可能让两个进程同时写同一个 persisted session。事故场景里，进程 A 已经基于某个 tail 记录了工具结果，进程 B 从旧 tail resume 后继续 append，两个写入都成功但 parent chain 分叉；之后从物理尾恢复会丢掉另一条已展示历史。

#7166 覆盖完整 single-writer 方案，但体量大。#7237 抽出可独立落地的 P0a：优先保护 ACP/daemon live writer、append fencing、close drain、pinned runtime/storage 和错误面，避免再次出现同类并发 writer 损坏。

## 当前实现方案

当前 open diff 为每个 `(runtime base, session ID)` 建立跨进程 writer lease。租约通过 atomic hard-link 协议选出唯一 owner；owner 获取后会重新加载权威 transcript 物理尾，再允许 ACP session 继续。每次 append 前校验 owner token、transcript file identity、metadata 与 UTF-8 byte length，只有校验通过才推进内存 tail。

daemon 会复用已 live 的 owner，而不是为同一 session 建第二个 writer。用户 prompt、cron、notification、teammate turns 都要先通过 ownership gate；live transcript replay 也穿过 owner barrier 和 pinned `SessionService`。关闭路径等待 active turns drain 并收到 acknowledged close 后才移除 live entry、释放 lease；close 卡住时不会提前释放 writer。

runtime root 和 persistence root 在逻辑 `cwd` 变化后仍 pinned 到原 session runtime，避免 ACP fallback roots、storage、extension manager 或 desktop metadata 写到错误目录。writer conflict 会通过稳定的 ACP/HTTP sanitized error 暴露给用户，desktop 不会把冲突的 persisted session 静默替换成 fresh session。
