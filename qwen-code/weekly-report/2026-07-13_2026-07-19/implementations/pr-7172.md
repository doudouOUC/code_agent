## 解决的问题

Plan mode 过去把 confirmation 形态当成 shell 安全性的代理，但 Shell 和 Monitor 都可能是只读、明确写入或 parser unknown。不同 approval host 对 unknown command、host rewrite、stale approval、abort race、persistent approval 和 IDE diff 的处理可能不一致，导致 unknown shell 越过 Plan mode 边界，或用户在狭窄确认视口里看不到足够上下文。

## 当前实现方案

当前 diff 在 Core 引入 `plan-mode-shell-policy.ts`，把模型发起的 Shell/Monitor 调用接入 #7053 的三态 classifier。`read-only` 命令继续走现有权限流程；明确 `write` 在任何审批界面授权前被拒绝；`unknown` 只能通过 exact one-time approval，且 Plan mode 保持 active。

exact approval 会 snapshot raw request、validated invocation、working directory、Plan-mode revision、approval policy 和 permission manager 结果。执行前重新校验这些上下文：Plan revision 变更、cwd 变更、permission policy 变更、raw input 改写、`updatedInput` 不一致、`newContent` 或未提供的 persistent option 都会转为 cancel。成功批准只向工具传空 payload，不生成持久 allow rule。

这套契约被传到 ACP `Session`、stream-json/dual-output permission controller、nested subagent、teammate、background agent 和 speculation gate。Monitor 使用 `normalizeMonitorCommand(command).safetyCommand` 分类；speculation 只有 `read-only` 才继续，write/unknown/parser failure/empty input 都停止。TUI compact confirmation 重新分配 warning 和 diff 的 body height，极短终端下优先保留 exact command、问题和一次性批准/取消选项。
