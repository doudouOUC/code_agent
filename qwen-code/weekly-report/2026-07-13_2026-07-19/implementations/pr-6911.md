## 解决的问题

workspace-qualified active session export 已经能从 selected trusted workspace 读取 active persisted JSONL，但 archived transcript 仍必须先 unarchive 才能导出。这样会改变 archive 状态，也会在 active/archived 同 id 或 primary/secondary 同 id 时引入错读风险。旧客户端如果只看到 `workspace_session_export`，也无法表达“我要 archived storage，而不是 active storage”。

## 最终实现方案

PR 新增独立 capability `workspace_archived_session_export` 和独立 route `GET /workspaces/:workspace/session/:id/archive/export?format=html|md|json|jsonl`。selector 先按 registered workspace id、再按 encoded canonical cwd 解析；selected runtime 必须 trusted。route 只读 selected workspace 的 `chats/archive/<id>.jsonl`，不扫描 active storage、不 fallback primary、不解析 live owner、不启动 ACP、不 attach client、不加载 settings。

错误语义与 archived storage 状态绑定：active-only id 返回 `409 session_not_archived`，missing 返回 `404 session_not_found`，active/archive 双写返回 `409 session_conflict`，archive/unarchive/delete 进行中返回 `409 session_archiving`。route 在 reconstruct 前执行 archived transcript 256 MiB indexing limit，超过返回 `413 transcript_too_large`；active export 不因此改变既有 no-cap contract。

core `SessionService.loadArchivedSession()` 复用 `loadSession` 的私有 reconstruction 逻辑，但固定读取 archive path。export route 持有既有 `SessionArchiveCoordinator` shared lease，覆盖 location check、transcript reconstruction 和 formatting；archive/unarchive/delete 继续持 exclusive lease，避免 transition 与 export 交错。formatter、attachment headers、MIME type、filename sanitization、cache/no-sniff headers 和 SDK attachment parser 都复用 active export。`WorkspaceDaemonClient.exportArchivedSession()` 强制 native REST transport，老 daemon 缺方法时返回普通 HTTP error。
