## 解决的问题

`GET /health?deep=1` 最初服务于 single-workspace daemon，只读 primary bridge 的 session、pending permission、active prompt 等计数。multi-workspace 落地后，secondary workspace 中存在 session/prompt/permission 时，deep health 仍可能返回空闲，影响外部 idle detection 和诊断。shallow `/health` 又必须保持便宜，只证明 listener 可响应，不能改成重型 readiness。

## 当前实现方案

当前 open diff 将 deep health 定义为 daemon-wide informational snapshot。route 从 `WorkspaceRegistry.listManaged()` 读取所有 managed runtime，包括正在 draining 但 bridge cleanup 尚未完成的 runtime；对每个 runtime 读取 required bridge getters 后聚合。`sessions`、`pendingPermissions`、`activePrompts` 求和；`workspaceCount` 是 runtime 数；`channelAlive` 表示任一 managed runtime channel live；`lastActivityAt` 取最新非空 activity；`idleSinceMs` 使用同一个 `Date.now()` snapshot 减去最新 activity；`connectedClients` 和 optional rate-limit hits 保持 daemon-wide counter。

deep health 不暴露 workspace id、path、trust state 或 per-workspace detail，也不 ping child process/channel，因此仍不是全 workspace readiness 或真实 liveness。`deep=1`、`deep=true` 和 bare `deep` 启用 deep path，其它值保持 shallow。单 workspace 响应保留原值并 additive 增加 `workspaceCount:1`。如果 registry 或任一 getter throw，整个 deep probe 返回 `503 {"status":"degraded"}`，不返回 partial snapshot。bootstrap listener ready 但 runtime registry 未 ready 时，也返回 degraded + `Retry-After: 1`，同时 health-first startup 仍会触发 runtime startup。
