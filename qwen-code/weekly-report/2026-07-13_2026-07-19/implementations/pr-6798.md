## 解决的问题

multi-workspace daemon 已经能把 live session 归属解析到 secondary runtime，但一批 legacy session action route 仍然沿用 primary-only guard 或直接调用 primary bridge。结果是可信 secondary session 的重命名、recap、BTW、mid-turn message、task cancel 和 goal clear 会返回 `non_primary_session_route_not_supported`，或者存在误分发到 primary runtime 的风险。

## 最终实现方案

PR 把 6 条既有 route 改为 owner-aware dispatch：`PATCH /session/:id/metadata`、`POST /session/:id/recap`、`POST /session/:id/btw`、`POST /session/:id/mid-turn-message`、`POST /session/:id/tasks/:taskId/cancel` 和 `POST /session/:id/goal/clear` 从 `withMutableSession` 切到 `withOwnerMutableSession`。route 仍先按原逻辑做参数校验、client id 解析、abort controller 和错误映射，真正调用 bridge 方法时改为 `runtime.bridge.*`，其中 `runtime` 由 live session owner resolver 选出。

该实现没有新增 URL、capability、payload 或 SDK API，主要是修正路由分发边界。metadata 仍截断 `displayName` 到 256 字符；BTW 仍保留问题长度限制、socket close abort 和 `AbortError` 静默返回；task cancel、goal clear、mid-turn message 的 response shape 不变。untrusted secondary owner 继续在 owner resolver 阶段返回 403，unknown owner 返回 404，不回退到 primary。

## 关键代码路径

- `packages/cli/src/serve/routes/session.ts`: 6 条 session action route 改用 `withOwnerMutableSession` 并调用 owning `runtime.bridge`。
- `packages/cli/src/serve/multi-workspace-sessions.test.ts`: 增加 secondary-only dispatch、primary array empty、untrusted/unknown/invalid input 覆盖。
- `packages/cli/src/serve/server.test.ts`: 覆盖 BTW 等 route 的 legacy 行为不回归。
- `docs/developers/qwen-serve-protocol.md`: 标注这些 singular legacy route 已按 live owner runtime 分发。

## 验证方式

- `cd packages/cli && npx vitest run src/serve/multi-workspace-sessions.test.ts src/serve/server.test.ts`
- 手工 smoke：双 trusted workspace daemon 中创建 secondary live session，分别调用 metadata、recap、BTW、mid-turn、task cancel、goal clear，确认请求只进入 secondary bridge，primary bridge 没有调用记录。
