## 解决的问题

cold first session 的耗时横跨 bootstrap deferred runtime、channel preheat/worker spawn、ACP initialize、ACP child `newSession()`、settings/config/auth/file-system/session registration 和最终 `GeminiClient.startChat()`。旧的 daemon route span 只能看到 runtime app 接手后的 HTTP duration，浏览器看到的 `POST /session` 时间也无法拆出 proxy、runtime wait、channel wait 和 child bootstrap，导致首个 session 慢时只能猜。

## 最终实现方案

最终实现沿用现有 OTel 体系，不新增 response header、public JSON 字段或 capability。bootstrap delegating app 在 runtime 未挂载时记录 request arrival、runtime wait 和 `started_on_request | joined` path，并把 timing 挂到 Express request；真正的 daemon telemetry middleware 后续读取该 timing，回填 HTTP span 起点和 request duration metrics，使 deferred runtime wait 不再被误认为网络/代理时间。

ACP bridge 在 `spawnOrAttach()` 前同步分类 channel path：已有可用 channel 是 `reused`，已有 in-flight spawn 是 `joined`，否则是 `spawned_on_request`。随后用 `channel.wait` span 包住 `ensureChannel()`。每个 `ChannelInfo` 生成 daemon-owned diagnostic UUID，附到 `channel.spawn`、`channel.initialize` 和后续 `session.new` span 上，用于把自动 preheat trace 与后来的 session trace 相关联，但不伪造 parent/child 因果。

`session.new` span 内向 ACP `NewSessionRequest._meta` 注入 daemon traceparent。ACP child 的 `QwenAgent.newSession()` 提取该 context，创建 `qwen-code.daemon.session_start` span，并记录固定 stage duration：`settings_load`、`config_setup`、`auth`、`file_system_setup`、`session_register`、`response_build`。失败时记录 `failed_stage` 但不改变原错误。已有 `QWEN_CODE_PROFILE_SESSION_START` JSONL profiler 增加可选 `sessionId`，用于与 trace 中的 ACP session id 做 join。
