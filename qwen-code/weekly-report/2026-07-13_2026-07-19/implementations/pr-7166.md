## 解决的问题

两个 Qwen Code 进程可以同时恢复同一个 session，从各自内存里的尾节点继续向同一个 JSONL transcript 追加记录。两个进程都可能成功返回，但 parent chain 已经分叉；重启时只能从物理文件尾重建其中一条分支，用户已经看到的工具结果、回答或后续上下文可能从恢复后的会话里消失。

单纯重启无法修复这种损坏，因为分叉已经写进文件。需要在写入前证明当前进程是唯一 writer，并在每次 append 前确认 transcript 没有被外部改动。

## 当前实现方案

当前 open diff 为每个 `(runtimeBase, sessionId)` 引入 `SessionWriterLease`。租约文件位于 runtime base 下的 `session-writer-locks` namespace，记录 owner id、pid、hostname、work dir、session id 和 acquire time。获取租约后才权威重载 transcript；每次 JSONL append 都校验当前 lock owner 与预期 UTF-8 byte length，发现 owner 丢失或 transcript 长度变化即进入永久 integrity failure，后续 model turn fail-closed。

stale local lock 可在确认 PID 不存活后通过 rename-to-stale 再 reclaim；foreign-host lock 不自动回收。`SessionService` 的 resume、clear、rename、delete、archive、unarchive、fork、branch 和 offline maintenance 都进入同一 lease namespace。live session 被另一个 owner 持有时返回结构化 `session_writer_conflict`；offline 维护会获取临时 lease 并在完成后释放。

会话切换路径改成 commit-before-repoint：`/resume`、`/clear`、branch、cwd relocation 等在新 Core/UI state 成功提交前保留旧 recorder 和旧 lease，失败时旧会话继续可用。daemon close、auth failure、restore failure、ACP detach、desktop/VS Code lifecycle 和 process shutdown 只释放 owner 匹配的 lease 与 runtime sidecar，防止旧回调清掉新 owner。

运行状态与可观测性同步增强：runtime status sidecar 带 owner id，clear/deactivate 需要 owner 匹配；`qwen-code.session.writer.event.count` 统计 conflict、stale reclaim、writer lost、transcript changed；ACP/HTTP 错误映射暴露冲突但不泄露本地进程细节。#7237 进一步从该方案中抽出可独立落地的 ACP/daemon P0a writer fence，优先保护 live owner reuse、append fencing、close drain 和 pinned runtime roots。
