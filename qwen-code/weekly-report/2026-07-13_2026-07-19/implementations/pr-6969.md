## 解决的问题

daemon file logging 过去按 PID / workspace hash 生成文件，正常重启和 crash/restart 会持续堆积；外部 logrotate 如果 rename/truncate active file，会破坏 daemon 的写入模型。并发 daemon、startup failure、fallback logging、queue overflow、append unknown、close timeout 和 access-log 洪泛都缺少统一的 bounded ownership 与 status surface。

## 当前实现方案

当前 open diff 将每个 runtime log namespace 收敛为稳定 active path `${runtimeBaseDir}/debug/daemon/daemon.log`。正常重启 append 到同一路径；healthy stable family active file 10 MiB，保留 4 个 archives，单条 file record 最多 256 KiB，accepted-but-unsettled file payload 最多 4 MiB。每次启动生成 immutable 128-bit `runId`，file record 强制带 runId/PID，caller context 不能覆盖。

stable writer 持 lifetime `proper-lockfile` lease。抢不到 stable lease 的并发 daemon 写入 `debug/daemon/runs/run-<runId>/daemon.log` fallback family，并持 `.owner.lock`；它不会在本 run 中 promote 到 stable。fallback allocation/cleanup 用 `runs/.maintenance.lock` 串行化，保留所有 busy owner family 和最多一个 inactive family。`QWEN_DAEMON_LOG_FILE=0|false|off|no` 直接返回 healthy stderr-only logger，不碰 filesystem。

file write 进入 Promise queue 保序。超过 4 MiB pending cap 的 record 只丢 file copy，stderr 不截断；logger 累计 dropped records/bytes，并在 capacity 恢复后写 file-only loss summary。append reject 视为结果未知，标记 `write_failed` 并停止后续 file mutation，不把失败 record 算作精确 loss。lease compromise 同样停止新 file mutation。rotation 在写入会超过 10 MiB 前验证 archive dir、prune 到 3 个旧 archive、生成唯一 archive name、atomic rename active，再写新 active；rotation 失败会 drop 当前 record，并限速重试，不让 active file 越界。

daemon status summary/full 增加 logger snapshot：`daemon.runId`、`logMode`、`logHealth`；full 另含 `logPath`、`logIssues`、`logDroppedRecords`、`logDroppedBytes`。degraded logging 添加 path-free `daemon_log_degraded` warning。access log 增加 constant-space token bucket（burst 60，2 records/s），健康/heartbeat/成功 SSE 排除保持不变；无 token 时只累积 2xx/3xx/4xx/5xx/other 五类 counter，恢复时输出 `access logs suppressed` summary。
