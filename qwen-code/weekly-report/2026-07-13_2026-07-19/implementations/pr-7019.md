## 解决的问题

multi-workspace daemon 已经从早期 “1 daemon = 1 workspace” 演进成一个 daemon 托管多个 isolated workspace runtimes，但设计文档、用户指南、命令帮助、协议表和部分代码注释仍残留旧前提。#6378 hardening 也缺少可执行的收口标准：哪些 ownership/failure 语义是当前基线，哪些属于未来 feature，很容易在 review 中继续扩张。

## 当前实现方案

当前 open diff 新增 `docs/design/daemon-multi-workspace-hardening.md`，把 daemon route 分成 process-global、legacy-primary、workspace-qualified、live-session-owner、persisted-workspace 五类，并固定 fail-closed 语义：unknown/untrusted/ambiguous/draining/removed 状态都不能 silently fallback primary。文档明确一个 daemon 仍只有 process-global token/rate limit/listener/fault radius，但每个 active workspace runtime 拥有自己的 env snapshot、bridge、workspace service、route filesystem、session metadata、channel worker group 和 MCP pool/budget scope。

用户和开发者文档同步把 superseded 的单 workspace 前提改成当前模型：repeatable `--workspace` 注册 isolated runtimes，first workspace 是 primary；trusted secondary lazy 启动自己的 ACP child；untrusted secondary 只允许明确的 read-only persisted surface；daemon-managed named channels 按 owning trusted workspace 分组，`--channel all` 保持 primary-only；workspace mismatch 文案改为“该 bridge 不 fallback primary”。同时为 protocol 文档里的 conditional serve capabilities 加 `<!-- conditional-serve-features:start/end -->` marker，并新增 `capabilities-docs-contract.test.ts`，要求文档表与 `CONDITIONAL_SERVE_FEATURES` 精确一致，覆盖当前 27 个 tag。
