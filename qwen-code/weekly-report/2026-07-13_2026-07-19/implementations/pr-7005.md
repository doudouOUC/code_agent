## 解决的问题

multi-workspace daemon 里 branch、fork、`cd` 三条 live-session mutation 仍是刻意 primary-only，但保护它们的 wrapper 名称和错误文案带有 Phase 2a 临时感，测试覆盖也没有证明 primary/secondary bridge 都不会被误调用。同时 serve 代码里对 `process.env` 的整文件放行过宽，workspace-scoped 逻辑可能绕过 review 直接读进程级环境；`workspace/setup-github` 的 proxy fallback 就存在应使用 owning runtime env 却可能读 primary/process env 的风险。

## 最终实现方案

最终实现把 primary-only live-session mutation 收敛成显式 closed contract：`PRIMARY_ONLY_LIVE_SESSION_ROUTES` 精确列出 `POST /session/:id/branch`、`POST /session/:id/fork`、`POST /session/:id/cd`，`withPrimaryOnlyLiveSession()` 只接受这组 route，未知 route 会启动期/测试期 fail-loud。secondary owner 请求继续返回 HTTP 400 `non_primary_session_route_not_supported`，响应 shape 保持 route、sessionId、workspaceId、workspaceCwd，但错误文案改成长期可维护的“only available for primary workspace sessions”。拒绝时写 `daemonLog.warn('session routing failed', ...)`，且不调用 primary 或 secondary bridge。

环境隔离方面，`process-env-guard.test.ts` 从整文件 exception 改为 AST 级 allowlist：每个直接 `process.env` 读取必须匹配具体 key 或 access 表达式，并附带 process-scoped 理由。`workspace/setup-github` 的 `resolveSetupGithubProxy()` 改为接收 runtime env；REST route、ACP Streamable HTTP dispatcher 和 workspace-qualified ACP mount 都把 owning runtime 的 effective env 传入。`runtimeEffectiveEnv()` 对空 runtime overlay 返回空对象，防止空 overlay 继续继承 daemon env。嵌入场景保留 daemon-level fallback。

## 验证方式

- `packages/cli/src/serve/multi-workspace-sessions.test.ts`: secondary branch/fork/cd 全部返回 `non_primary_session_route_not_supported`，两个 runtime bridge 都不执行。
- `packages/cli/src/serve/process-env-guard.test.ts`: 新增直接 `process.env` 读取必须被精确 allow 或删除。
- `packages/cli/src/serve/routes/workspace-setup-github.test.ts`: REST workspace setup 使用 owning runtime proxy。
- `packages/cli/src/serve/acp-http/transport.test.ts` 与 `workspace-qualified-acp.test.ts`: ACP setup-github、CDP MCP runtime config、empty overlay isolation。
- 手工 smoke：双 trusted workspace 中对 secondary live session 调 branch/fork/cd，确认响应指向 secondary workspace 且 primary 无 fallback。
