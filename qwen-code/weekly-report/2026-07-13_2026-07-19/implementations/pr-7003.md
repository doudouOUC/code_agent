## 解决的问题

multi-workspace legacy session routes 已逐步改成 owner-aware，但 daemon telemetry middleware 在 handler 之前做路由分类，无法只靠 singular `/session/:id/...` URL 判断真实 workspace runtime。旧实现会让部分 legacy route 没有 canonical request span，或把 rewind/shell/owner-routed route 归因到 primary workspace；如果 middleware 和 handler 都解析 live owner，还会重复扫描 owner，增加漂移和竞态面。

## 当前实现方案

当前 open diff 新增声明式 `legacySessionTelemetryRoutes` catalog，覆盖 48 条显式 `/session`、`/sessions` 与 `/permission` legacy route。每个 entry 固定 method、path、canonical route label 和归因类型：41 条 `handler_resolved` 等待 handler 发布实际 runtime，7 条 `pre_resolved` 仍按 primary-bound 兼容面在 middleware 阶段归因。`resolveDaemonTelemetryRoute()` 改为先查 catalog，再处理 workspace-qualified route、daemon status 等非 legacy route，并对 workspace transcript/export/archive export 的 session id 做 decode。

handler-resolved route 在 middleware 里先不开 `qwen-code.workspace.hash`，而是在 response 上挂 telemetry context；`requireSessionRuntime()`、session creation、load/resume 和 legacy transcript resolver 在选出唯一 runtime 后调用 `setDaemonTelemetryWorkspace(res, runtime.workspaceCwd)`。请求 finish/close 时 middleware 再 hash 该 workspace 并写入 span 属性。未解析、歧义、workspace mismatch 或 not found 的请求不写 workspace hash，也不 fallback primary。request metrics 继续保持 daemon-global，长连接 session event stream 有 span 但不记普通 request latency；heartbeat 和 daemon status 仍排除 dashboard metrics。
