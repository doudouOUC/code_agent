## 解决的问题

daemon/ACP 客户端可以让一个 session 长时间存活。模型在自然 stop 前如果刚写了 unfinished top-level Todo list，当前请求会被视作完成，但实际任务可能只是暂时停在可自动继续的中间态。旧行为只能依赖外部 Stop hooks 或用户再次提示，daemon 本身没有 bounded、内建、可观察的 “Todo 未完成则继续当前任务” 机制。

## 当前实现方案

当前 open diff 新增 opt-in `experimental.todoStopGuard`，默认 `false`，需要 restart，且 safe mode、bare mode、Approval `plan` mode 强制关闭。它不是外部 hook，因此不受 `disableAllHooks` 影响；但 permission、cancellation、token/loop protection、external Stop hook cap、daemon resource limit 仍然更高优先级。

CLI `Session` 持有 `DaemonTodoStopGuard` in-memory 状态机，记录当前 work chain 是否 armed、unfinished todo count、已提交 continuation attempts、是否 suspended/awaiting FIFO prompt，以及 exhaustion 是否已报告。只有成功执行的 top-level `TodoWriteTool.execute()` 且返回 structured `{ type:'todo_list', todos:[...] }` result 才能 arm guard。参数、replay/disk state、失败/重复 tool call、subagent Todo list 和 shadowed `todo_write` wire name 都不可信。空 todo 或全部 completed 会立即 disarm。

自然 stop 时执行固定顺序：先 drain mid-turn user message；再让 daemon FIFO 中完整且未 abort 的 prompt 优先；再执行已有 external Stop hooks；最后在 guard armed、未 suspended、未 awaiting queued prompt、有 unfinished items、没有相关 background input 时触发 guard continuation。如果 external Stop hook 和 guard 同时阻止 stop，则合并 reason 到同一次 continuation call。每个 uninterrupted automatic continuation stage 最多两次 primary-model stream；mid-turn user message 开启新的 two-attempt stage，retry/continue 和相关 background result 保留当前 budget。

guard 会 snapshot work-chain 开始时的 background agents、shells、monitors、wakeups 和 queued notifications。只有新出现且相关的 background input 才阻塞 guard continuation；旧 task notification 或 recurring cron 会延后到 active chain 不能再 resume 后再独立运行。rewind/history restore、branch/fork、cwd change、new Session、disk restoration、daemon/agent restart 会清除 trust；live attach、model change 或非 Plan approval mode change 不单独清除。

每次 guard continuation 和最终 exhaustion 都作为 replayable `session_update` 发出，`_meta.source` 为 `todo_stop_guard`，metadata 只包含 attempt 和 unfinished count，不包含 Todo 文本。queued full prompt、permission/cancel、PostToolUse shouldStop、loop protection、token limits 等路径会 suspend 或终止 guard，避免自动继续越权。
