## 解决的问题

cold daemon channel startup 里，父进程只能看到一个 `channel.initialize` 总耗时。这个边界里实际包含 child process/module load、Gemini import、bootstrap config、tool/config 初始化、initialize handler 和 response transport。缺少分阶段数据时，后续 P0 启动优化只能猜测瓶颈，容易改 readiness、timeout 或 Session 语义。

## 最终实现方案

最终实现新增一个 opt-in ACP initialize metadata handshake。父进程只有在 initialize request 里带 `_meta.qwen.daemon.channelStartupProfile.v = 1` 时才请求 profile；旧父进程或未协商请求不会收到 profile，新子进程仍兼容旧父进程。

ACP child 在导入 Gemini 前启用轻量 `acp-startup-profiler`，记录固定 startup marks 和 bootstrap config events。profile 构建后冻结为 v1 schema，只包含固定 phase/config key 和 bounded duration。父进程通过 `channel-startup-profile.ts` 校验 profile 版本、字段名、数值范围和 10 分钟上限，校验失败、缺字段、telemetry disabled 或解析异常都 fail-open，不影响 initialize 成功。

父进程把校验后的 timing 写入既有 `channel.initialize` span，属性前缀为 `qwen-code.daemon.acp_startup.*`。`scripts/check-serve-fast-path-bundle.js` 也新增 guard，确保 `qwen serve` pre-listen bundle 不静态加载 ACP/Gemini profiling runtime，避免观测能力反过来拖慢 daemon 启动。

- `packages/acp-bridge/src/bridgeTypes.ts`: `CHANNEL_STARTUP_PROFILE_META_KEY`、version 和 `ChannelStartupProfileV1` wire schema。
- `packages/acp-bridge/src/channel-startup-profile.ts`: parent-side profile validation、duration bounds 和 span attribute flattening。
- `packages/acp-bridge/src/bridge.ts`: initialize request negotiation、response profile parsing、`channel.initialize` span attribute injection。
- `packages/cli/src/utils/acp-startup-profiler.ts`: child-side marks、config event aggregation、profile freeze。
- `packages/cli/src/acp-integration/acpAgent.ts`, `cli.ts`, `gemini.tsx`: profiler bootstrap before Gemini import、initialize response `_meta` wiring。
- `packages/core/src/config/config.ts`, `daemon-tracing.ts`: config startup event sink 与 explicit daemon parent context。
- `scripts/check-serve-fast-path-bundle.js`: pre-listen static closure guard。
