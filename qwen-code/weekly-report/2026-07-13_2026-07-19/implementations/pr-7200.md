## 解决的问题

primary workspace 的 ACP status/preheat routes 已存在，但 released client 无法通过 `/capabilities` 独立发现它们；SDK 也可能把 REST-only control-plane 调用误走 ACP transport。并发 preheat 时，短 timeout waiter 还可能清掉共享 channel initialization single-flight，导致服务端仍在启动 child，客户端却拿到模糊失败。

multi-workspace 场景下，如果客户端在 secondary workspace 上盲调 primary ACP preheat/status，会误触 primary runtime。preheat 响应里的 `ready`、duration、timeout 和错误净化也缺少稳定契约。

## 最终实现方案

最终实现新增两个 v1 capability：`workspace_acp_status` 和 `workspace_acp_preheat`。它们只描述 primary-workspace ACP status/preheat control-plane surface；客户端必须 feature-detect 且确认 selected workspace 精确等于 primary workspace 后才能调用。Web UI 缺 capability 或选中 secondary workspace 时不会探测或预热。

preheat route 定义 point-in-time readiness：`ready === channelLive`，duration 使用 monotonic integer milliseconds。`timeoutMs` 如果 present 但非法会返回 400，不再静默回退默认值。客户端可见错误被净化，daemon log 保留详细 cause；短 HTTP waiter timeout 不会清理 service-level single-flight，后续 waiter 仍可复用同一次 channel 初始化。

TypeScript SDK 把 ACP status/preheat 强制走 native REST transport，即使 `DaemonClient` 配置了 ACP HTTP/WS transport，也不会把 control-plane 调用路由到 session ACP。docs 同步记录 capability versioning、serve protocol 和 preheat contract。
