## 解决的问题

shell safety 旧接口只有 boolean：要么 `read-only`，要么不是。这个模型无法区分“静态已证明会写”与“静态无法证明安全”。Plan mode 的后续收口需要把 unknown shell command 单独交给一次性审批，而不是把 parser 失败、wrapper、env prefix、substitution、pager/helper、动态执行等不确定面和明确写操作混成同一类。

旧 scheduler 还会先 unwrap shell wrapper 再判断同步只读，导致 `bash -c 'git status'` 这类 wrapper 可能被当作只读并发执行。权限系统和调度层需要一个更保守的事实层：只有确证只读才走只读路径；明确写操作标 write；其他不确定情况保留 unknown，给上层策略继续做 fail-closed 或用户审批。

## 当前实现方案

当前 open diff 在 `shellAstParser.ts` 中新增 `classifyShellCommandSafety(command)`，返回 `read-only`、`write`、`unknown` 三态，并统一使用 `write > unknown > read-only` 的合并优先级。AST 里出现 `ERROR` 节点直接归 unknown；command/process substitution 会把外层结果抬到 unknown floor，但 substitution 内部发现明确 writer 时仍提升为 write；control flow 扫描所有可能分支，function definition 本身不当作执行 body；纯 assignment 和 `cd` 保持既有兼容行为。

parser failure 的语义也被拆清：三态 public API 在 tree-sitter lazy load、parse runtime exception 或 private classifier throw 时返回 unknown，不再用 regex fallback 伪造确定性。旧 boolean API `isShellCommandReadOnlyAST()` 仍保留兼容 fallback：只有三态结果为 `read-only` 才返回 true；parser 无法加载或抛错时仍使用原 regex 只读判断；但语法错误 AST 属于正常 parse 结果，会以 unknown/false 返回，不再 fallback。

规则实现被抽到新的 `shell-safety-rules.ts`。它集中维护 bounded sed/awk scanner、shell glob helper、direct writer 判定和 option parser，避免用脆弱 regex 误把文件名或 message 当成 write/help。明确写操作包括文件/进程状态修改、output redirection、Git mutation family、`find`/`sed`/`awk`/`sort`/`tree`/`uniq`/`tee`/`dd` 中带写语义的参数；动态执行、外部脚本、解释器/wrapper、pager、git signature/textconv/helper、ripgrep preprocessor、hostname helper、archive search helper、不支持的 git global option 和大小写不匹配命令都归 unknown。

调度层同步只读判断也更保守：`CoreToolScheduler` 不再先 `stripShellWrapper(command)` 再调用 sync read-only checker，而是把原始命令交给 checker。这样 `git log`、`ls` 等直接只读命令仍可并发，`bash -c 'git status'` 这类 wrapper 会排到顺序队列，`sort -o output input`、`npm install` 等明确写/unknown 继续阻断后续 shell。

权限 manager 当前仍消费 boolean API，所以 #7053 本身不改变 default permission routing，也不直接实现 Plan mode unknown 一次性审批。它把事实层准备好：后续策略可以读取三态结果，把 unknown 和 write 分开处理。

## 关键代码路径

- `packages/core/src/utils/shellAstParser.ts`: 三态 classifier、AST traversal、parser failure/ERROR 语义、boolean compatibility wrapper。
- `packages/core/src/utils/shell-safety-rules.ts`: sed/awk scanner、shell pattern helper、git/find/sed/awk/sort/tree/uniq/tee/dd 等规则。
- `packages/core/src/utils/shellReadOnlyChecker.ts`: 同步 checker 与三态/boolean API 的衔接。
- `packages/core/src/core/coreToolScheduler.ts`: shell sync-readonly 调度改为检查原始命令，不再 unwrap wrapper。
- `packages/core/src/permissions/permission-manager.ts`: default shell permission 继续用 boolean 兼容 API。
- `docs/design/shell-safety-classification.md`: read-only/write/unknown 判定边界、fail-closed 策略和非目标范围。
