## 解决的问题

#6740 的 workspace-qualified persisted transcript reader 会被 untrusted secondary workspace 使用，前端通常是在收到并解析 JSON 后才做过滤/渲染。仅靠 record-count `limit` 不能限制一条巨大 aggregate record 或一个巨大 replay cursor 带来的内存和响应膨胀。daemon 需要在源记录读取、响应序列化和 cursor 编解码三个层面加硬边界。

## 最终实现方案

PR 保持 legacy singular `GET /session/:id/transcript` 的 cursor key、ACP error mapping 和分页语义不变，只收紧 workspace-qualified route。默认 page size 仍为 100，最大 500，snapshot 256 MiB guard、32-entry/64 MiB index cache 和五分钟 cache lifetime 继续沿用。

workspace-qualified route 新增三个 byte budget：每页最多读取 4 MiB persisted source records，serialized response JSON 最多 32 MiB，request/response cursor state 最多 64 KiB。`limit` 因此只是 record-count ceiling，不再承诺成功页面一定返回这么多 record；读取会在 record 边界停止，避免把 partial aggregate record 切坏。

如果单个 aggregate record 超过 4 MiB source budget，或者 serialized response 超过 32 MiB，route 返回 `413 transcript_page_too_large`。如果 replay cursor 自身过大，服务端把已经转换出的内容作为 terminal partial page 返回，并不再发 continuation cursor，避免客户端拿到一个必然被拒绝的下一页 cursor。
