## 解决的问题

session JSONL 的 `parentUuid` 链如果中间断一环，旧 `SessionService` 会从最新 leaf 往前回溯并在缺失 parent 处停止，最终只恢复断点之后的 tail island。终端 `/resume` 和 ACP/Web Shell replay 都会静默显示这段 tail，用户看不到“更早历史已丢失”的提示。

读侧不能安全把更早 island 拼回来：缺失 parent 既可能来自存储损坏，也可能来自丢失的 `/rewind` marker。贸然恢复 earlier island 会复活用户已经 rewound away 的内容。因此最终方案是“检测并标记断点”，而不是恢复丢失段。

## 最终实现

核心新增 `packages/core/src/utils/conversation-chain.ts`，提供 `buildOrderedUuidChain(records, { leafUuid, detectGaps })`。它继续沿 `parentUuid` 从 leaf 回溯；当 parent 缺失时，默认保持旧截断行为，只在 `detectGaps` 打开时返回 `HistoryGap { childUuid, missingParentUuid }`。早期 patch 里尝试过跨 island bridge，但最终 review 后移除，避免恢复不安全历史。

`SessionService.reconstructHistory()` 使用该 helper，并在交互恢复路径返回 `historyGaps`。终端 `/resume` 的 `resumeHistoryUtils` 和 ACP replay 的 `HistoryReplayer` 都按 `childUuid` 索引 gap，在对应 child record 前插入本地化的 info/agent message，提示 earlier conversation lost and could not be recovered。两端共用 `history-gap-notice.ts` 的 formatter 和 `indexGapsByChild()`，保证 wording 与 keying 一致。

background-agent resume 路径显式不打开 gap detection：它没有能展示 gap marker 的交互 surface，只检测后丢弃会造成“半检测”。该路径继续保持历史行为，在断链处截断。`resumeHistoryUtils` 还在 gap divider 处清空 pending `@` 命令状态，避免 pre-gap `@file` 被错误配到 post-gap 用户消息。

## 关键代码路径

- `packages/core/src/utils/conversation-chain.ts`: `buildOrderedUuidChain()`、`HistoryGap` 类型、detect-only 缺口记录。
- `packages/core/src/services/sessionService.ts`: reconstruct history 时返回 `historyGaps`，并透传给 load/resume 调用者。
- `packages/cli/src/ui/utils/history-gap-notice.ts`: 本地化 notice formatter 和 gap index helper。
- `packages/cli/src/ui/utils/resumeHistoryUtils.ts`: `/resume` 历史项构建时插入 info divider，并在 divider 处清空 pending `@` 状态。
- `packages/cli/src/acp-integration/session/HistoryReplayer.ts`: ACP/Web Shell replay 在 gap child 前发 agent message notice。
- `packages/cli/src/acp-integration/acpAgent.ts`: `loadSession` / transcript replay 透传 `historyGaps`。
- `packages/core/src/agents/background-agent-resume.ts`: 继续只截断，不做无 UI 的 gap detection。
