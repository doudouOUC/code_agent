## 解决的问题

#6409 已阻止大型 PDF 全文 `pdftotext` 结果直接进入上下文，但 dense PDF、扫描 PDF 或没有文本层的 PDF 仍会卡在文本提取失败/超预算提示上。对配置了 vision 模型的环境来说，这些 PDF 可以用 page image 读取，旧实现却没有 bounded image fallback，导致模型反复缩小页码仍拿不到内容。

## 最终实现

该 PR 在 PDF 读取链路增加 `pdftoppm` page-to-JPEG fallback。默认仍优先走 `pdftotext`，只有 text extraction 失败或超过 `PDF_TEXT_RESULT_MAX_TOKENS` 时才进入 render path；text-only 主模型但配置了 vision bridge 时，`@` 附件和 path read 可以把扫描页渲染成图片交给 vision 模型转写。

`packages/core/src/utils/pdf.ts` 新增 `isPdftoppmAvailable()` 缓存探测、`renderPDFPagesToImages()` 和渲染常量。渲染命令使用 `pdftoppm -jpeg -scale-to 1600`，不和 `-r` 混用，保证每页最长边固定、base64 与 vision token 成本有界。单次 render 总 base64 上限为 25 MiB，120s timeout；临时目录用 `mkdtemp` 创建并 best-effort 清理。password/corrupt/invalid/timeout/no-output 会返回明确错误。

`processSingleFileContent()` 把 PDF fallback 排成有序链：显式 text extraction 成功则返回文本；文本失败/超预算时，若可 render 且有 vision 消费路径则返回 image parts；如果没有 `pages` 且只渲染了前 N 页，会追加文本提示，说明后续页被 page cap 或 byte cap 省略，要求用户用 `pages` 指定范围。`read_file`、`read_many_files` 和 `pathReader` 共享该策略。

## 关键代码路径

- `packages/core/src/utils/pdf.ts`: `pdftoppm` availability cache、JPEG 渲染、page filename 排序、byte cap、timeout/error mapping。
- `packages/core/src/utils/fileUtils.ts`: PDF text extraction overflow/failure 后的 image fallback、page omission notice、vision parts 输出。
- `packages/core/src/tools/read-file.ts`: read_file PDF fallback 接线。
- `packages/core/src/utils/readManyFiles.ts`、`pathReader.ts`: 多文件/路径读取复用 PDF image fallback。
- `packages/core/src/utils/pdf.test.ts`、`fileUtils.test.ts`: render、fallback 顺序、vision bridge 与 omission notice 覆盖。

## 验证方式

- `cd packages/core && npx vitest run src/utils/pdf.test.ts src/utils/fileUtils.test.ts src/tools/read-file.test.ts src/utils/readManyFiles.test.ts src/utils/pathReader.test.ts`
- 手工验证：扫描 PDF 或 dense PDF 在 vision 模型可用时返回 image parts；缺少 `pdftoppm` 时给出安装 poppler-utils 的提示；无 `pages` 的长文档不会静默丢页。
