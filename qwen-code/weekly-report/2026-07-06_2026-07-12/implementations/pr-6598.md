## 解决的问题

`qwen serve --channel` 托管的 channel worker 启动后只在进程启动时读取 settings。渠道 token、proxy、per-channel model 等配置变更后，旧方案需要重启整个 daemon 才能生效，这会影响 daemon 内已有 live sessions。需要一个只重启 channel worker、不拆 daemon 的 reload 面。

## 最终实现

该 PR 给 daemon-managed channel worker 增加显式 reload。`ChannelWorkerSupervisor` 新增 `restart()`：先 `stop()` 当前 worker，再 `start()` relaunch，使新 worker 重新读取 settings；并发 reload 会 coalesce 到同一个 stop+start promise。`killAllSync()` 会 latch `disposed`，reload 与 daemon teardown 竞争时不会 fork 孤儿 worker；显式 reload 会重置 restart budget，让已经进入 failed/parked 状态的 worker 可以由 operator 恢复。

对外暴露三层入口：HTTP `POST /workspace/channel/reload`、SDK `DaemonClient.reloadChannelWorker()`、CLI `qwen channel reload`。HTTP route 只有当 `getChannelWorkerSnapshot` 与 `reloadChannelWorker` 两个 deps 都被 wire 时才注册，并走 strict mutation auth；worker 未启用时返回 `409 channel_worker_not_enabled`；成功返回 `{ reloaded:true, worker:<snapshot> }`。能力标签 `channel_reload` 也只在这两个 deps 同时存在时广告，避免 capability 显示但 route 404。

CLI 命令支持 `--daemon-url` / `--token`，并回退 `QWEN_DAEMON_URL` / `QWEN_SERVER_TOKEN`，默认 URL 为 `http://127.0.0.1:4170`。成功后输出重载后的 worker snapshot，失败走 stderr 并设置非零退出码。
