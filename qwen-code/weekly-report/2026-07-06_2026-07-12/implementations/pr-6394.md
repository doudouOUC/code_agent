## 解决的问题

daemon multi-workspace 后续阶段需要一个内部 runtime registry，把每个 workspace 的 bridge、workspace service、REST route filesystem factory 和 client-MCP sender registry 聚合到统一边界。但当前 `qwen serve` 仍是单 workspace 装配，相关对象通过 server assembly 和 `app.locals` 散落传递。重复显式 `--workspace` 输入也会落入旧单 workspace 路径，缺少明确失败语义。

## 最终实现

该 PR 落地 Phase 1 single-runtime registry，不启用 public multi-workspace 行为。

`packages/cli/src/serve/workspace-registry.ts` 新增 `WorkspaceRuntime` 和 `WorkspaceRegistry`。`createServeApp()` 仍先构建现有 primary bridge/workspace service/fsFactory/client-MCP registry，再包装成 `workspaceRegistry.primary`，并保留 `app.locals.fsFactory`、`boundWorkspace` 等 legacy route locals，保证现有路由 schema、SDK types、`/capabilities` 和单 workspace 行为不变。

同时，daemon identity 从 workspace-scoped 改为 daemon-scoped：`daemon-logger.ts` 使用 `daemon:<pid>` 和 `serve-<pid>.log`，workspace hash 只作为 log/telemetry metadata；telemetry middleware 按 request-time workspace cwd 计算 workspace hash，为后续 workspace-qualified route 留边界。`serve` 命令和 `run-qwen-serve.ts` 识别 yargs 可能给出的 `workspace: string[]`，单元素数组按 primary workspace 处理，多个显式 `--workspace` 在 runtime boot 前报错。

## 关键代码路径

- `packages/cli/src/serve/workspace-registry.ts`: `WorkspaceRuntime` / `WorkspaceRegistry`、primary/list/exact lookup。
- `packages/cli/src/serve/server.ts`: server assembly 通过 registry primary 传递 workspace runtime，保留 legacy locals。
- `packages/cli/src/serve/run-qwen-serve.ts`: 单 workspace array 接受、多 workspace array boot guard、daemon logger/telemetry wiring。
- `packages/cli/src/commands/serve.ts`: yargs `--workspace` repeatable shape 处理。
- `packages/cli/src/serve/daemon-logger.ts`: daemon-scoped id/log file，workspace hash 作为 metadata。
- `packages/cli/src/serve/server/telemetry.ts`: daemon-scoped service instance id 与 request-time workspace hash。
- `.qwen/design/daemon-multi-workspace-phase1-registry.md`: Phase 1 边界和非目标。
