## 解决的问题

chat recording 的 JSONL append 失败原本只在 debug 路径可见，recorder 还可能继续推进 parentUuid。这样后续记录会接在一个没有落盘的 parent 后面，恢复时形成断链或静默丢历史。`recordCustomTitle()` 也会在 durable append 前返回成功，导致 UI 显示 title 已改但 JSONL 中没有对应记录。对于 daemon/WebUI/ACP/structured output 用户，这类失败既不可见，也无法可靠阻止后续损坏。

## 最终实现方案

`ChatRecordingService` 的 write chain 在第一次实际 JSONL append rejection 后会进入 terminal failure state。失败会保留原始 rejected write chain，后续 `flush()` 仍能暴露同一个写失败；同时 recorder 停止接受新的 descendant record，避免在缺失 parent 后继续写出错误链。这个停止是持久的，不会因为后续 append 成功而恢复。

custom title 改成 durable 语义：`recordCustomTitle()` 只有在有序 append 成功后才返回成功，显式 rename 的优先级高于 auto-title。ACP 调用侧等待 durable result，但不强行等待 unrelated trailing flush；`/branch` 在切换 recorder 和 UI 前会持久化并 reload fork title，保证 fork 后 UI title 与 JSONL title 一致。

失败通知走一个通用 `recording_stopped` 事件/通知路径。interactive TUI、headless non-interactive、structured output、ACP bridge、daemon SSE、TypeScript SDK normalizer 和 WebUI provider 都能收到可见 warning。payload 只带 failed session id 和通用停止语义，不暴露本地 filesystem path、errno 或内部异常栈；支持多个 listener，并通过 EventBus snapshot 让通知不因短期 eviction 丢失。

## 关键代码路径

- `packages/core/src/services/chatRecordingService.ts`: terminal failure state、write chain rejection 保存、durable custom title。
- `packages/core/src/config/config.ts`: recording failure listener 注册与多 listener 分发。
- `packages/cli/src/utils/chat-recording-failure.ts`: 通用 warning/事件构造与可见通知。
- `packages/acp-bridge/src/bridge.ts`, `bridgeClient.ts`: recording failure event/snapshot 传播。
- `packages/cli/src/acp-integration/session/Session.ts`, `acpAgent.ts`: ACP title/recording durable result 处理。
- `packages/cli/src/ui/AppContainer.tsx`, `renameCommand.ts`, `useBranchCommand.ts`: TUI 可见 warning、rename/branch durable flow。
- `packages/cli/src/nonInteractiveCli.ts`, `nonInteractive/session.ts`: headless 与 structured output warning。
- `packages/sdk-typescript/src/daemon/events.ts`, `ui/normalizer.ts`: `recording_stopped` known event 与 UI signal。
- `packages/webui/src/daemon/session/DaemonSessionProvider.tsx`: WebUI provider 订阅并显示 recording stopped signal。
