## 解决的问题

#6625/#6716 让 daemon 可以在运行时新增并持久化 secondary workspace，但缺少对应的热卸载能力。用户删除 persistent registration 只会影响下次启动，当前 daemon 里对应 runtime、sessions、ACP mount、remember lane、memory task、channel worker 和 path reservation 都仍然存在。长期运行的 Web Shell 因此无法释放误加 workspace，也无法重新添加同一路径。

## 当前实现方案

当前 open diff 新增 `workspace_runtime_removal` 能力与 `DELETE /workspaces/:workspace` route。capabilities 的 workspace rows 增加可选 `removable` 字段，只有动态/可移除 secondary runtime 才能热删除；primary workspace 和显式启动时配置的 static workspace 均不可删除。

删除流程是同步两阶段 drain。非 force 请求会先做 fast pre-drain activity snapshot，发现 live session、pending prompt/start、ACP connection、memory task、sub-session launcher、channel worker 或 registration in-flight 时返回 `409 workspace_busy`。可以 drain 时，route 先关闭 admission、ACP、worker 等 gate，防止新请求进入，再采集 final activity snapshot，随后清理 selected runtime：关闭 workspace-owned sessions、pending starts、ACP dispatcher/connection registry、remember/memory lane、sub-session launcher、bridge child、channel worker，并从 registry 释放 workspace path。

持久化层会删除该 runtime 的所有 known registration alias，但不会删除项目文件、settings、transcripts、archive 或 sidecar。`force:true` 允许越过 busy activity 阻断，但仍保留“不删除用户文件”的边界。若 persistence 或 cleanup 失败，route 返回结构化错误如 `workspace_persist_failed` / `workspace_runtime_removal_failed`；非 force drain 检测到 activity 时会回滚 admission/ACP/worker gates。

SDK/WebUI/Web Shell 增加 remove workspace flow：先根据 capabilities 检查 `workspace_runtime_removal` 和 `workspace.removable`，遇到 busy snapshot 时提示用户确认 force，再发送 force delete。UI 删除后刷新 capabilities/workspace list，而不是假定本地状态已经同步。
