## 解决的问题

Phase 3 已把 core REST routes 扩展为 `/workspaces/:workspace/...`，但 ACP transport 仍只有 legacy `/acp`，绑定 primary workspace。ACP-native 客户端若连接 multi-workspace daemon，无法直接打开 secondary workspace 的 ACP session；反向 WebSocket、remember lane、client MCP sender 和连接诊断也没有 per-runtime 边界。

## 最终实现

该 PR 在 legacy `/acp` 旁增加 workspace-qualified ACP endpoint：`/workspaces/:workspace/acp`。HTTP POST、GET SSE 和 DELETE 都通过同一套 shared handler，但在进入 handler 前先解析 workspace selector；primary selector 复用 primary ACP mount，trusted non-primary selector 使用该 runtime 自己的 bridge、workspace service、fsFactory、client-MCP sender registry 和 `WorkspaceRememberTaskLane` 创建独立 ACP dispatcher 与 `ConnectionRegistry`。unknown selector 返回 `workspace_mismatch`，untrusted non-primary workspace 返回 `untrusted_workspace`。

WebSocket upgrade 仍只有一个 server listener，但 path routing 增加 `/workspaces/<selector>/acp` 形态。listener 使用 raw request target 做 selector 解析，在 Express/URL 规范化前拒绝 dot-segment、反斜杠和异常编码路径，避免 `/workspaces/%2e%2e/acp` 这类路径被 normalize 后误落 primary。共享安全检查之后，listener 按 workspace id 或 encoded cwd 解析 runtime，把 initialize、session events、prompt dispatch、connection registry 与 reverse client-MCP provider 都切到对应 runtime。CDP tunnel 仍 gated on primary mount；non-primary 不承诺 workspace-scoped CDP。

OAuth device-flow registry 最终保持 daemon-global/shared，因为 OAuth credential 本身是进程级资源。变化点不是 per-runtime registry，而是 auth-flow event fanout：device-flow 事件会 best-effort 分发到 primary 和 trusted secondary bridge，一个 bridge 失败不阻塞其他 bridge。`AcpHttpHandle.getSnapshot()` 聚合所有 mount 的 connection/stream 计数和诊断，并在 full diagnostics 里标注 workspace id/cwd/primary；dispose 后 HTTP handler 统一返回 `503 server_disposed`，且 dispose/attachServer 生命周期保持 terminal/idempotent。

能力注册新增 `workspace_qualified_acp`，但只在 ACP HTTP enabled 且 multi-workspace sessions enabled 时广告。legacy `/acp` 保持 primary workspace 兼容；`/daemon/status` 和 rate-limit/runtime issue 聚合改成跨 mounts 统计。
