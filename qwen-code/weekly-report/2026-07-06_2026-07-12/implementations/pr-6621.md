## 解决的问题

Phase 3 已把 core REST routes 扩展为 `/workspaces/:workspace/...`，但 ACP transport 仍只有 legacy `/acp`，绑定 primary workspace。ACP-native 客户端若连接 multi-workspace daemon，无法直接打开 secondary workspace 的 ACP session；反向 WebSocket、device flow、remember lane 和 client MCP sender 也没有 per-runtime 边界。

## 当前实现方案

当前 open diff 在 legacy `/acp` 旁增加 workspace-qualified ACP endpoint：`/workspaces/:workspace/acp`。HTTP POST、GET SSE 和 DELETE 都通过同一套 shared handler，但在进入 handler 前先解析 workspace selector；primary selector 复用 primary ACP mount，trusted non-primary selector 使用该 runtime 自己的 bridge、workspace service、fsFactory、client-MCP sender registry、device-flow registry 和 `WorkspaceRememberTaskLane` 创建独立 ACP dispatcher 与 `ConnectionRegistry`。unknown selector 返回 `workspace_mismatch`，untrusted non-primary workspace 返回 `untrusted_workspace`。

WebSocket upgrade 仍只有一个 server listener，但 path routing 增加 `/workspaces/<selector>/acp` 形态。共享安全检查之后，listener 按 workspace id 或 encoded cwd 解析 runtime，把 initialize、session events、prompt dispatch、connection registry 与 reverse client-MCP provider 都切到对应 runtime。CDP bridge runtime-MCP wiring 保持 primary-only；non-primary mount 的 CDP hooks 是 no-op。

`workspace-registry.ts` 为 runtime 增加 optional `deviceFlowRegistry`，`device-flow-registry.ts` 拆出 `createDeviceFlowRegistry()`，让 non-primary runtime 有自己的 OAuth device-flow 状态，而不是写入 app-global primary registry。能力注册新增 workspace-qualified ACP tag，仅在 multi-workspace daemon 中广告；design doc 记录 Phase 4 边界。
