## 解决的问题

#6567/#6631 已让 trusted secondary workspace 通过 workspace-qualified route 查看 session list、session groups、organized/archived view，并能在 selected workspace 下创建/管理 group。但实际给 session 设置 pin、group 或 color 的 mutation 仍只有 legacy `PATCH /session/:id/organization`，它固定绑定 primary workspace。结果是客户端能看到 secondary organization state，也能创建 secondary group，却不能把 secondary session 放进该 group 或修改其 pinned/color 状态。

## 最终实现方案

该 PR 新增 `PATCH /workspaces/:workspace/session/:id/organization`。route 先用 workspace id 或 encoded canonical cwd 解析 selected runtime，并要求 runtime trusted；legacy `PATCH /session/:id/organization` 保持 primary-only 兼容行为。两条 route 共用 `handleSessionOrganizationUpdate()`，但 target 中显式传入 `workspaceCwd`、`bridge` 和 route name，避免 mutation 路径隐式回落到 primary。

mutation 仍走现有 `mutate()` gate、`SessionArchiveCoordinator.runSharedMany([sessionId])` 并复用原请求/响应类型。存在性检查改为 selected workspace 作用域：先用 `new SessionService(target.workspaceCwd).sessionExistsInAnyState(sessionId)` 查 active/archived persisted session；不存在时再用 `target.bridge.getSessionSummary(sessionId)` 验证 live session 是否属于同一 workspace。非空 `groupId` 也由 selected workspace 的 `SessionOrganizationService(target.workspaceCwd)` 校验；unknown group id 返回既有 `group_not_found`。这样不会通过全局 session-owner 猜测跨 runtime 修改，也不会把 secondary session 错写进 primary sidecar。

SDK 在 `WorkspaceDaemonClient` 上新增 `updateSessionOrganization(sessionId, update, clientId?)`，生成 `/workspaces/:workspace/session/:id/organization` 请求。协议文档同步说明：客户端要同时 pre-flight `session_organization` 和 `workspace_qualified_rest_core`；legacy mutation 仍 primary-only。
