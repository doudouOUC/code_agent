## 解决的问题

`qwen serve --channel` 的 daemon-managed channel worker 原先是 daemon-wide 单 worker，默认跟 primary workspace runtime/env/settings 绑定。multi-workspace daemon 中，workspace-scoped channel 配置可能属于 secondary workspace；如果仍由 primary worker 读取，就会使用错误 cwd、settings、env 或 webhook 配置。状态面、pidfile 和 reload API 也只能表达一个 worker，无法诊断每个 workspace 的 channel worker。

## 当前实现方案

当前 open diff 增加 channel-to-workspace grouping。启动时 `resolveChannelWorkspaceGroups()` 按 selected channel 名解析 owner：显式 cwd 或 workspace-scoped channel config 归属对应 trusted workspace；用户/system scope 且无 cwd 的 channel 视为 ambiguous；配置 cwd 未注册返回 mismatch；untrusted workspace fail fast。`--channel all` 暂保持 primary-only v1，并输出提示，避免在自动展开所有 workspace channel 前改变既有语义。

每个 trusted owner workspace 启动一个 `ChannelWorkerSupervisor`，并把 worker 绑定到该 runtime 的 `workspaceCwd`、`QWEN_DAEMON_WORKSPACE` 和 `runtime.env.effectiveEnv`。webhook config 也从 owning workspace 读取，webhook dispatch 按 channel owner 路由到对应 worker。

新增 `ChannelWorkerGroup` 管理多个 supervisors：启动阶段顺序 start，失败时回滚已启动 worker；`stop()` 等待所有 restart 完成后停止；`restart()` 是 daemon-wide fail-closed 事务，顺序重启所有 selected workspace workers，任一 worker restart 失败则停止整个 group，避免混合 generation 继续对外服务。并发 restart 会 coalesce；webhook route 找不到 owner/supervisor 时返回 `channel_worker_unavailable`。

兼容面保持：单 workspace 时不输出 `workers[]`，旧 top-level `channels` 是 union，`workerPid` 仍是 primary worker pid。multi-workspace 时 pidfile `ServiceInfo.workers[]` 记录 `workspaceId/workspaceCwd/channels/workerPid`；`/daemon/status.runtime.channelWorkers[]` 暴露每个 workspace worker snapshot，旧 `channelWorker` 字段指 primary 或首个 worker。SDK 类型新增 `DaemonChannelWorkerGroupSnapshot`，reload response 仍返回 primary/first snapshot，完整列表建议读 `/daemon/status`。
