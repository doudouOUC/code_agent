## 解决的问题

#6625 让 Web Shell 可以在 daemon 运行时通过 `POST /workspaces` 注册 secondary workspace，但这些 runtime 只存在于进程内。daemon 重启后，用户必须重新添加 workspace，workspace-qualified REST/ACP/Web Shell 也无法恢复到之前的多工作区形态。需要一个 user-level、按 primary workspace 隔离的持久化 desired-state，同时不能把工作区列表写进项目目录或一次性信任所有历史路径。

## 最终实现方案

已合入 `WorkspaceRegistrationStore`。store 文件位于 `${QWEN_HOME:-~/.qwen}/daemon/workspaces/<primary-scope-sha256>.json`，schema v1 保存 canonical primary workspace 与 secondary workspace path 列表；scope hash 在 Windows 下对 primary path 做小写归一。读取时拒绝非普通文件、symlink、超过 256 KiB、malformed JSON、schema 不匹配、primary 不匹配、重复路径、primary 被写成 secondary，以及超过 24 个 secondary workspace 的 store。

写入路径同时做进程内 promise chain 和 `proper-lockfile` 跨进程锁，锁目录以 `0700` 创建；store 通过临时文件、`0600` 权限、rename 和目录 fsync 做原子写。`WorkspaceRegistrationStore.add()` 去重返回是否新增，`removeById()` 使用 `workspaceRegistrationId(cwd)` 的 16 字节 hash 作为 public id，避免 URL 暴露完整路径。

`runQwenServe()` 默认启用 store，除非 `QWEN_SERVE_NO_PERSISTENT_REGISTRATION=1`。启动时先解析显式 `--workspace`，再读取持久化列表，逐条 canonicalize、跳过与显式/已恢复 workspace 重复或嵌套的条目，并受 `MAX_REGISTERED_WORKSPACES=25` 总上限约束；store 读取失败只打 stderr，daemon 继续用显式 workspace 启动。恢复出来的 workspace 走与显式 secondary workspace 相同的 runtime startup path。

`POST /workspaces` 新增 `persist` 布尔参数。`persist:false` 保持 #6625 的进程内注册行为；`persist:true` 时要求 store 可用，禁止持久化 primary，允许把已存在的 non-primary runtime 补登记到 store。新 runtime 场景先创建 runtime，再写 store，再 `workspaceRegistry.add(runtime)`；如果 registry add 失败且 store 已写入，会按 id 回滚持久记录并 shutdown 未使用 runtime。persist 写失败返回 `workspace_registration_store_error`，runtime 创建失败仍返回 `runtime_creation_failed`。

新增 `GET /workspace-registrations` 返回当前 store snapshot、entry id、cwd、active 与 persisted；新增 `DELETE /workspace-registrations/:id` 只删除持久化 desired-state，不卸载已经运行的 runtime，响应用 `restartRequired: active` 提醒该 runtime 要到下次重启才消失。`persistent_workspace_registration` capability 仅在 `workspaceRegistrationStore` 注入/可用时广告；SDK `DaemonClient.addWorkspace(cwd, { persist })` 会在 body 中发送 `persist:true` 并读取可选 `persisted` 字段。Web Shell 的 add workspace UI 增加 persist 选项与 i18n 文案。

workspace-qualified ACP mount 也随这次改动调整为 lazy：`createServeApp()` 不再只在 app 创建时根据 workspace 数量预判是否启用 plural ACP path，而是在 ACP HTTP enabled 时挂好 `/workspaces/:workspace/acp` route，让后续动态注册或启动恢复的 workspace 也能通过同一路由解析到新 runtime；capability 仍只有在 multi-workspace sessions 可用时才有语义。

## 关键代码路径

- `packages/cli/src/serve/workspace-registration-store.ts`: 持久化 store schema、scope hash、跨进程锁、原子读写、去重和按 id 删除。
- `packages/cli/src/serve/run-qwen-serve.ts`: 启动读取 persisted workspaces、跳过无效/嵌套/超限条目，并把 store 注入 `createServeApp()`。
- `packages/cli/src/serve/routes/workspace-management.ts`: `POST /workspaces { persist }`、`GET /workspace-registrations`、`DELETE /workspace-registrations/:id` 与失败回滚。
- `packages/cli/src/serve/capabilities.ts`: 条件能力 `persistent_workspace_registration`。
- `packages/cli/src/serve/server.ts`: capabilities 使用动态 workspaceRegistry 计数，ACP plural mount 支持动态 workspace。
- `packages/sdk-typescript/src/daemon/DaemonClient.ts`: `addWorkspace(cwd, { persist })` 和 `persisted` response 字段。
- `packages/web-shell/client/components/sidebar/WebShellSidebar.tsx`: Web Shell add workspace 时支持持久化注册。
