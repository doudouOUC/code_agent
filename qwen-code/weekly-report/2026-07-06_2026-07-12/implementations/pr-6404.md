## 解决的问题

`read_file` 和 `@file` 读取纯文本日志时遇到 10MB 以上文件会直接返回 `FILE_TOO_LARGE`，模型无法看到任何有效上下文。CI log、测试输出等大文本产物常常只需要局部行范围；硬拒绝会阻断排障，而整文件读入又有内存和输出风险。

## 最终实现

当前 open diff 新增大文本范围读取链路。`packages/core/src/utils/readTextRange.ts` 提供小文件/大文件分流：小文本保留原有 encoding-aware read；大文本按 `line`、`limit`、`maxOutputBytes` 流式扫描，返回有界内容、总行数、换行/编码 metadata 和 `truncatedByBytes`。非文本、图片、音频、视频、PDF 和 binary size guard 保持旧保护。

core 读取接口扩展为 `CoreReadTextFileRequest`，增加 0-based `line`、`maxOutputBytes` 和 `AbortSignal`。`StandardFileSystemService.readTextFile()`、`readFileWithLineAndLimit()`、`read_many_files`、`pathReader`、`read_file` 和 `@file` 处理链路都传递 signal/limit metadata。ACP 边界由 `AcpFileSystemService` 把 core 0-based line 转成 ACP 1-based line，并只把协议支持的字段发给远端；fallback 本地读则保留 core-only 字段。

默认 `read_file` 对大文本不再硬报 10MB，而是返回带 `Showing lines X-Y of N total lines` 和 `... [truncated]` 的有界内容；显式 offset/limit 可以读取指定行段。

## 关键代码路径

- `packages/core/src/utils/readTextRange.ts`: 大文本范围读取、byte truncation、line metadata 和 cancellation。
- `packages/core/src/utils/fileUtils.ts`: `readFileWithLineAndLimit()` 接入大文本 range 逻辑。
- `packages/core/src/services/fileSystemService.ts`: `CoreReadTextFileRequest`、`maxOutputBytes`、`signal` 和 `_meta.truncatedByBytes`。
- `packages/core/src/tools/read-file.ts`: `execute(signal)` 传递取消信号。
- `packages/core/src/utils/readManyFiles.ts`、`pathReader.ts`: 多文件/路径读取链路接入 range metadata。
- `packages/cli/src/acp-integration/service/filesystem.ts`: core request 到 ACP readTextFile request 的 0-based/1-based 转换。
- `packages/cli/src/ui/hooks/atCommandProcessor.test.ts`: `@file` 大文本截断读取覆盖。
