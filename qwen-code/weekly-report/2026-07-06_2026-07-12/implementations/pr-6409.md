## 解决的问题

纯文本模型不支持原生 PDF input 时，qwen-code 会通过 `pdftotext` 做文本 fallback。旧逻辑允许大型 PDF 全文提取，100 页级 PDF 可能产出约 100k 字符并进入 prompt，即使自动压缩后也可能触发 hard prompt safety limit。另一方面，用户仍需要能读大 PDF 的局部页码，不能简单恢复 10MB 硬拒绝。

## 最终实现

PDF 读取被拆成三层预算。

第一层是全文 fallback gate。`processSingleFileContent()` 对 PDF 先解析 `pages`，再判断是否会走 text extraction。没有 `pages` 且模型需要文本 fallback 时，先调用 `getPDFPageCount()`；拿不到页数时用 `PDF_PAGE_COUNT_SIZE_HEURISTIC_BYTES` 估算页数。`shouldRequirePDFPageRange()` 判定超过 `PDF_FULL_TEXT_PAGE_LIMIT=10` 后返回短 guidance，提示使用 `pages` 读取具体范围。direct `read_file` 默认把它作为 `FILE_TOO_LARGE`；`read_many_files` / `@` 附件通过 `largePdfBehavior:'reference'` 把它作为 lightweight reference 放进上下文，不算失败读。

第二层是 pdftotext 输入上限。全文 fallback 仍有 `PDF_FULL_TEXT_EXTRACTION_MAX_MB=100`；显式页码读取放宽到 `PDF_PAGED_TEXT_EXTRACTION_MAX_MB=512`，保证大型文档可以抽样读页，但不会把任意超大文件交给 poppler 长时间处理。

第三层是输出 token guard。`extractPDFText()` 成功后，`estimatePDFTextOutputTokens()` 复用 request tokenizer 的 ASCII/CJK 估算逻辑，并加 wrapper token margin；超过 `PDF_TEXT_RESULT_MAX_TOKENS=12000` 时，`buildPDFTextTooLargeGuidance()` 返回更窄 page range 建议。单页过密时不会建议继续缩页，而是提示换 native PDF-capable model、外部分割或使用其它工具抽更小片段。

## 关键代码路径

- `packages/core/src/utils/pdf.ts`: PDF page range policy、token estimate、guidance builders、统一 `PDF_TEXT_EXTRACTION_UNAVAILABLE_MESSAGE`。
- `packages/core/src/utils/fileUtils.ts`: `largePdfBehavior`、full/paged extraction size cap、page-count gate、output token guard。
- `packages/core/src/tools/read-file.ts`: `pages` 描述改为使用 `PDF_MAX_PAGES_PER_READ` 常量，并说明大型 PDF 需要 page range。
- `packages/core/src/utils/readManyFiles.ts`: `@` 附件路径对大型 PDF 使用 reference 行为。
- `packages/core/src/utils/request-tokenizer/textTokenizer.ts`: 抽出 `estimateTextTokens()` 供 PDF 输出 guard 复用。
- `packages/core/src/utils/*.test.ts` 与 `packages/core/src/tools/read-file.test.ts`: 覆盖 direct read、attachment reference、page extraction、dense output、pdfinfo unavailable 等路径。
