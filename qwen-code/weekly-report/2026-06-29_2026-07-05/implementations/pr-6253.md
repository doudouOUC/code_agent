## 解决的问题

`GET /daemon/status` 已经提供了较完整的 daemon 运行态 JSON，但排障时仍需要 curl + jq 手工阅读。operator 需要一个可直接打开的只读 dashboard 快速查看 daemon 状态、session 容量、transport、workspace section、ACP connection、auth device flow、rate-limit 和安全配置；Web Shell 用户也需要在同一 UI 内打开相同状态面。

## 最终实现

PR 新增两套只读 dashboard surface。

第一套是 `packages/cli/src/serve/dashboard.ts:getDashboardHtml()`，按既有 `/demo` 模式生成无外部依赖的 inline HTML/CSS/JS 页面，页面请求 `/daemon/status?detail=summary|full`，支持 Summary/Full 切换、自动刷新、手动刷新和 bearer token 输入缓存。`routes/health-demo.ts` 注册 `GET /dashboard`，loopback 上与 `/health`、`/demo` 一样可在 pre-auth 阶段访问；非 loopback 或 `--require-auth` 场景仍挂在 bearer auth 之后，并设置 `X-Frame-Options: DENY` 与 CSP `frame-ancestors 'none'`。

第二套是 Web Shell 内的 React dialog：`DashboardDialog.tsx` 定义 daemon status response 形状、fetch/auto-refresh 逻辑、summary/full 展示和 workspace section 折叠面板；`WebShellSidebar.tsx` 在 footer 增加 Dashboard 按钮；`localCommands.ts` 增加 `/dashboard` 本地命令；`App.tsx` 接入 dialog state 与命令处理；`i18n.tsx` 补英文文案。

## 关键代码路径

- `packages/cli/src/serve/dashboard.ts`: standalone `/dashboard` inline page。
- `packages/cli/src/serve/routes/health-demo.ts`: `/dashboard` 路由注册、loopback/pre-auth 与安全响应头。
- `packages/cli/src/serve/rate-limit.ts`: 把 dashboard 纳入 loopback HTML 页面豁免口径。
- `packages/web-shell/client/components/dialogs/DashboardDialog.tsx`: Web Shell dashboard dialog、status fetch、summary/full UI。
- `packages/web-shell/client/components/sidebar/WebShellSidebar.tsx`: sidebar footer Dashboard 入口。
- `packages/web-shell/client/constants/localCommands.ts`: `/dashboard` 本地 slash command。
- `docs/users/qwen-serve.md` / `docs/developers/qwen-serve-protocol.md`: 用户与协议文档中的 dashboard 入口说明。

## 验证方式

- `packages/cli/src/serve/server.test.ts -t "GET /dashboard"`: loopback 返回 HTML、loopback token 场景免 Authorization、non-loopback 需 bearer、跨 Origin 拒绝、anti-clickjacking header。
