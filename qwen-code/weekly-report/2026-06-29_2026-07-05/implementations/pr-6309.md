## 解决的问题

大型历史会话执行 daemon `session/load` 时，ACP child 会把历史帧逐条通过 child-to-daemon notification 推入 live fanout 路径。这样会在 load 期间制造大量 event-loop 工作、污染 reconnect ring，并可能干扰当前 subscriber。daemon 自己发起的 restore 更适合拿到一次性的 replay snapshot，但直接 ACP client 仍需要保持 streamed replay 兼容。

## 最终实现

`packages/acp-bridge` 增加 response-mode replay 请求选项和 Qwen 私有 `_meta`。bridge 发起 `loadSession` 时可以要求 child 在 JSON-RPC response 中携带 replay updates；bridge 解析后会 strip 私有 metadata，以 ACP 标准结果对外呈现。REST load 改用 response-mode：bridge 用 replay payload seed 当前 session snapshot，不通知 subscribers，也不写入 reconnect ring，REST response 继续返回既有 replay snapshot 字段。

默认 ACP 行为保持 streamed replay。`acpAgent.ts` 只有在请求 metadata 明确要求时才把 replay updates 放进 response；否则仍逐条发送历史 `session/update`。ACP Streamable HTTP 的 `/acp` load 结果仍是标准 response，session stream 初次 attach 时从 bridge snapshot 重放 replayed `session/update` frames，并支持 `Last-Event-ID` 对这批初始 replay 做确定性续传。

## 关键代码路径

- `packages/acp-bridge/src/bridge.ts`: response-mode replay option、metadata strip、snapshot seed 和 stream attach replay。
- `packages/acp-bridge/src/bridgeClient.ts`: load request metadata 与 replay response 解析。
- `packages/acp-bridge/src/bridgeTypes.ts`: replay option / metadata 类型。
- `packages/cli/src/acp-integration/acpAgent.ts`: ACP child 侧按请求模式返回 replay updates 或保持 streamed replay。
- `packages/cli/src/acp-integration/session/Session.ts`: load replay metadata 传递。
- `packages/cli/src/serve/acp-http/connection-registry.ts` / `dispatch.ts`: ACP HTTP binding 与 stream replay 行为。
- `packages/cli/src/serve/routes/session.ts`: REST load 使用 response-mode replay。
