## 解决的问题

`qwen serve` daemon 与 ACP child 之间已经记录 pipe message byte histogram，但 metric 只有 direction，不能判断 256 KiB 以上大帧来自 live `session/update`、bulk replay response、`loadUpdates` response，还是普通 JSON-RPC request/response。没有真实大帧归因时，后续 sidecar / frame cap 设计只能靠猜阈值和 payload 家族。

这个 PR 只解决 measurement：不能改变 pipe payload、frame limit、EventBus、SDK、公开协议、CLI flags、HTTP query 参数或 capabilities。

## 最终实现

`packages/acp-bridge/src/ndJsonStream.ts` 的 hooks 增加 `onMessageObserved({direction, bytes, message})`。读写方向都只在 JSON message 成功收发后调用 observer；原有 `onMessageSent/onMessageReceived` 仍只接收 bytes，保持现有 byte metrics 调用形态。所有 hook 异常都被吞掉，避免观测逻辑破坏 transport。

daemon 侧新增 `large-pipe-frame-observer.ts`，默认只处理 `bytes >= 256 KiB` 的 frame。observer 对已解析 message 做浅层分类，输出低敏字段：direction、bytes、threshold、messageKind、method、sourceClass、updateCount、采样策略、session update 类型、tool name/provenance、raw output kind 和若干文本/JSON 近似 byte 上限。它不会记录 payload、session id、client id、路径、prompt 或 raw tool output。

日志和 telemetry 都有限速：每 daemon 进程 60 秒最多 50 条 large-frame 记录，超过的 suppressed count 会挂到下一条记录上。`run-qwen-serve.ts` 在 ACP child pipe hooks 中把 `onMessageObserved` 接到 observer，同时保留原 pipe stats、histogram 和 status counter 行为。

## 关键代码路径

- `packages/acp-bridge/src/ndJsonStream.ts`: `NdJsonMessageObservation`、`onMessageObserved` hook 和 hook failure isolation。
- `packages/cli/src/serve/large-pipe-frame-observer.ts`: 大帧 threshold、source classification、低敏字段抽取、采样/限速与 telemetry/log 写入。
- `packages/cli/src/serve/run-qwen-serve.ts`: daemon pipe hooks 接入 observer，并继续记录原有 pipe stats。
- `.qwen/design/2026-07-05-large-frame-handling-measurement.md`: measurement-only 设计、source classes 和 sidecar 下一阶段候选。
