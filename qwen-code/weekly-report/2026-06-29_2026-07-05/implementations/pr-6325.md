## 解决的问题

多客户端集成默认复用 `sessionScope: "single"` 时，多个 prompt 会进入同一个 per-session FIFO。此前 `/daemon/status.runtime.activity` 只有 active prompt 数量，operator 只能看到正在执行的 prompt，看不到已经被 daemon 接收但还在队列里等待的 prompt，容易把共享 session 排队误判成 daemon 卡住。

同时已有 OTel prompt queue wait metric 没有进入 `/daemon/status.runtime.perf`，dashboard/SDK 不能从 status snapshot 直接看到本进程 FIFO 等待时间。

## 最终实现

`packages/cli/src/serve/daemon-status.ts` 在 `runtime.activity` 里新增 `pendingPrompts` 和 `queuedPrompts`。构建 status response 时遍历 bridge session snapshot：`pendingPrompts` 汇总每个 session 的 `pendingPromptCount`，`queuedPrompts` 默认用 `pendingPromptCount - active` 推导；如果 bridge 暴露 daemon-wide `pendingPromptTotal`，则优先使用它作为 queued prompt 数，避免不同 bridge 实现下重复推导。

`run-qwen-serve.ts` 维护 prompt queue wait 累计统计，并把 daemon telemetry 的 `promptQueueWait(durationMs)` 同步记录到 status perf snapshot 和 metrics ring。`DaemonPerfSnapshot` 因此新增 `promptQueueWait.count/meanMs/maxMs/lastMs`，而 queued prompts 也进入 `DaemonMetricsBucket` 的 gauge 维度。queued prompt 只作为 backpressure 信号，不会生成 warning/error issue。

协议与消费侧同步更新：`docs/developers/qwen-serve-protocol.md` 说明默认 `single` session 会共享 FIFO，独立对话应显式使用 `sessionScope: "thread"`；TypeScript SDK status report 类型增加 optional activity/perf 字段；Web Shell status dialog 的 i18n 文案从 active-only 调整为 pending/queued 口径。
