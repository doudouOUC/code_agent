## 解决的问题

`qwen serve` 的 ACP child 是长驻进程，但每次 `session/new`、`session/load` 和 resume 都同步重跑完整 `loadSettings(cwd)`：读取多个 settings 作用域、migration、JSON parse、structured clone、`${VAR}` 插值和 `.env` 向上搜索。这些重复 I/O 和解析阻塞同一个 event loop；#6292 修复并发 settings 实例错配后，调用点已经适合改成请求级缓存读取。

## 最终实现

新增 `packages/cli/src/config/settings-cache.ts`。缓存以解析后的 workspace directory 为 key，LRU 上限 64 个 workspace；命中时重新计算 fingerprint，只有四个 settings 文件 stat、发现到的 `.env` 文件列表和 stat、IDE trust state、`realpath(cwd)`、`realpath(os.homedir())` 全部一致才返回同一个 `LoadedSettings` 实例。settings 文件编辑、自身 `setValue()` 临时文件 rename 写入、`.env` 出现/删除/修改、`QWEN_HOME` / HOME 切换、IDE trust 变化都会失效。

fingerprint 采集 fail-open：遇到意外 fs 错误直接走完整 `loadSettings()`，失败结果不缓存。`environment.ts` 导出 `findEnvFiles()`，让缓存复用现有 `.env` 发现语义。ACP 的三个 session 入口从 `loadSettings()` 切到 `loadSettingsCached()`；相关测试中把缓存 mock 成 passthrough，避免破坏按次 `mockReturnValue` 的既有单测结构。

## 关键代码路径

- `packages/cli/src/config/settings-cache.ts`: workspace LRU、fingerprint、stat invalidation 与 fail-open full-load fallback。
- `packages/cli/src/config/environment.ts`: 导出 `findEnvFiles()` 供缓存复用。
- `packages/cli/src/acp-integration/acpAgent.ts`: `newSession` / `loadSession` / `unstable_resumeSession` 使用 cached settings。
- `packages/cli/src/config/settings-cache.test.ts`: cache hit、失效触发、失败不缓存、LRU 等覆盖。
- `packages/cli/src/acp-integration/acpAgent*.test.ts`: session 入口 mock 调整。
