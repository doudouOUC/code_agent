## 解决的问题

#6098 已经让 daemon-managed channel worker 的日志进入有界 buffer 和 status snapshot，但 worker stderr 透传路径仍可能把 bearer token、API key、proxy URL credential、secret env value、JSON secret field 等敏感值写入 daemon stderr 或 daemon log file。ACP child stderr forwarder 也有同类风险。

## 最终实现

`packages/acp-bridge/src/logRedaction.ts` 新增 `redactLogCredentials()`，按顺序处理 Bearer/QQBot token、Authorization header、DingTalk access token header、常见 API key 前缀、GitHub/GitLab/Slack token、AWS access key、key/value secret assignment、复合 env secret、JSON-quoted secret field 和 URL-embedded credential。

`packages/acp-bridge/src/spawnChannel.ts` 的 `createStderrForwarder()` 在正常换行 flush 和 64 KiB forced truncation 两条路径都先脱敏再写 `process.stderr`，并把同一安全行通过 `onDiagnosticLine` 回调给 daemon logger。`packages/cli/src/serve/channel-worker-supervisor.ts` 将 daemon channel worker stderr 从继承终端改为 pipe，接入同样的 line-buffered redaction、buffer cap 和 try/catch 防护，让 worker stderr 同时进入 daemon terminal 与 daemon log file 时都只看到脱敏内容。

## 关键代码路径

- `packages/acp-bridge/src/logRedaction.ts`: credential pattern 集合与 `redactLogCredentials()`。
- `packages/acp-bridge/src/spawnChannel.ts`: ACP child stderr forwarder、64 KiB forced truncation、diagnostic callback。
- `packages/cli/src/serve/channel-worker-supervisor.ts`: daemon channel worker stderr pipe、redacted stream attach、daemon log forwarding。
- `packages/acp-bridge/package.json` / `packages/cli/vitest.config.ts`: 测试项目配置补齐。

## 验证方式

- `packages/acp-bridge/src/logRedaction.test.ts`: Bearer/QQBot/Authorization、API key、secret env、JSON secret、URL credential、DingTalk token 等脱敏矩阵。
- `packages/acp-bridge/src/spawnChannel.test.ts`: normal flush、forced truncation 与 `onDiagnosticLine` 都输出脱敏内容。
- `packages/cli/src/serve/channel-worker-supervisor.test.ts`: worker stderr pipe、buffer cap、日志回调和 redaction 集成。
