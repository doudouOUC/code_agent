## 解决的问题

`qwen channel start` 是 standalone channel 入口，不能随 `qwen serve` 一起由 daemon 统一托管。结果是 channel adapter 无法复用 daemon 的 session lifecycle、SDK client、status/stop 管理和 workspace 绑定，也无法让 `qwen serve --channel` 成为单进程启动 daemon + channel worker 的生产入口。

## 最终实现

PR 在 `packages/cli/src/commands/serve.ts` 和 `packages/cli/src/serve/run-qwen-serve.ts` 为 `qwen serve` 增加 repeatable `--channel <name>` 与 `--channel all`。serve fast path 先完成 HTTP listen 和 runtime ready，再由 `packages/cli/src/serve/channel-worker-supervisor.ts` fork internal CLI entry，执行 `packages/cli/src/commands/channel/daemon-worker.ts`。

worker 通过 `@qwen-code/sdk/daemon` 创建 `DaemonClient`，读取 `/capabilities` 校验 workspace，再用 `DaemonChannelBridge` 回连当前 daemon。`createDaemonSessionFactory` 强制 `sessionScope: 'thread'`，避免 channel 会话污染 daemon 的默认 shared single session；`SessionRouter` 仍负责 user/thread/single routing key、tool call dispatch 和 session cleanup。

管理面同步补齐 `packages/cli/src/commands/channel/pidfile.ts`、`status.ts`、`stop.ts`、`runtime.ts` 与 `/daemon/status`：pidfile 记录 `owner: serve`、daemon URL、workspace、requested channels、worker pid；`qwen channel status` 能展示 serve-owned worker；`qwen channel stop` 对 serve-owned worker 给出正确提示，避免误杀 daemon 之外的独立 worker。

文档更新覆盖 `docs/developers/channel-plugins.md`、`docs/developers/daemon/15-channel-adapters.md`、`docs/developers/qwen-serve-protocol.md`、`docs/users/features/channels/overview.md` 与 `docs/users/qwen-serve.md`。

## 关键代码路径

- `packages/cli/src/serve/channel-selection.ts`: 解析 `--channel` / `--channel all`，生成 worker selection。
- `packages/cli/src/serve/channel-worker-supervisor.ts`: fork worker、等待 ready、维护 snapshot、写 pidfile。
- `packages/cli/src/commands/channel/daemon-worker.ts`: 校验 daemon URL/workspace，加载 channel 配置，构造 `DaemonChannelBridge` 和 `SessionRouter`。
- `packages/channels/base/src/ChannelAgentBridge.ts`: adapter-facing bridge contract，承接 #5978 的抽象。
- `packages/cli/src/serve/daemon-status.ts` 与 `routes/daemon-status.ts`: `/daemon/status` 暴露 `runtime.channelWorker`。
