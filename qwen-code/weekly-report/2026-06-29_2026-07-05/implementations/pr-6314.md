## 解决的问题

EventBus 已有 per-subscriber live frame count cap，但少量超大 live events 可以在未达到帧数上限前占用大量 daemon heap。需要限制慢 SSE consumer 的 live serialized byte backlog，同时不破坏 replay frames、告警帧和终止帧的 `Last-Event-ID` 语义。

## 最终实现

`packages/acp-bridge/src/eventBus.ts` 为每个 subscriber 记录 live frame count 和 live serialized bytes。live event push 时按序列化后的 frame size 更新 byte backlog；默认 byte cap 为 2 MiB，daemon 内部拥有该预算，不向客户端暴露 query 参数。frame count warning/eviction 仍保留，byte 维度达到阈值时会发更丰富的 `slow_client_warning`，累计 live bytes 超预算时只驱逐该 subscriber，`client_evicted.reason` 为 `queue_bytes_overflow`。

`forcePush` 的 replay、warning、eviction frames 不计入 live count / live bytes，保持重连 replay 不被自身 backlog 挤爆。单个 oversized event 在空队列上仍允许通过，避免健康消费者因单帧过大被立即拒绝；真正需要驱逐的是积压的累计 live bytes。SDK typed event schema additive 增加 warning telemetry 字段和 `queue_bytes_overflow` terminal reason。
