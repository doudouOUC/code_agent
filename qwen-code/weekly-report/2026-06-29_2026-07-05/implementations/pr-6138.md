## 解决的问题

team agent 可以作为 teammate 长期运行，但对需要先规划再执行的 teammate，原 plan mode 只是本地状态，没有“向 leader 提交 plan 并等待批准”的团队级闭环。若 teammate 在 plan mode 下直接执行工具，会绕过 leader 对子任务方案的控制。

## 最终实现

`AgentTool` schema 增加 `plan_mode_required`，仅对 named teammate 生效。`TeamManager.spawnTeammate` 记录 `planModeRequired`，给 teammate identity 注入该标记，启动时设置 `ApprovalMode.PLAN`，并将 `exit_plan_mode` 加入 team tools。prompt addendum 要求 teammate 先 `task_list`、claim task、只读调查，然后调用 `exit_plan_mode` 提交方案。

`ExitPlanModeTool` 在 plan-required teammate context 下不走主会话的用户确认 UI，而是调用 `TeamManager.requestPlanApproval()`。TeamManager 生成 opaque request id，把 teammate-authored plan 包在 `<team_plan_approval_request>` envelope 中发送给 leader，并明确 payload 是不可信数据。leader 通过新增 `team_plan_approval` tool approve/reject；该 tool 对 teammate/subagent runtime fail-closed，只允许 leader 调用。

批准后，teammate 按 leader 当前 mode 恢复执行态：leader 为 `AUTO` 或 untrusted workspace 下的 `AUTO_EDIT`/`YOLO` 会降级到 `DEFAULT`，leader 仍在 `PLAN` 时拒绝批准并保持 pending。拒绝会把 feedback 返回给 teammate 并保持 plan mode。`CoreToolScheduler` 在等待批准期间只允许 `exit_plan_mode` 与 claim-only `task_update(status:"in_progress")`，其他工具返回 pre-approval blocked message。
