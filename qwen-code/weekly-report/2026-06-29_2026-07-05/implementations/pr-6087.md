## 解决的问题

subagent、team agent 和 workflow-dispatched agent 属于被调用者，不应该拥有主会话的 plan lifecycle。它们若能调用 `enter_plan_mode` / `exit_plan_mode`，会打断 caller 的 plan 状态，或把子任务计划误包装成“用户已批准主任务执行”的信号。只靠工具列表过滤也不够，因为 deferred tool search 或显式工具名可能重新暴露这些工具。

## 最终实现

新增 `packages/core/src/agents/runtime/subagent-plan-tool-policy.ts`，集中定义 `SUBAGENT_PLAN_LIFECYCLE_TOOLS`、`isSubagentLikeExecutionContext`、阻断消息和 tool result builder。判断来源覆盖 `agent-context` 与 team identity，能同时识别普通 subagent 和 team agent。

阻断分三层：

- AgentCore / WorkflowOrchestrator 在组装 subagent 工具集时从 wildcard、default 和显式工具名中移除 `enter_plan_mode` / `exit_plan_mode`。
- `tool-search.ts` 在 subagent context 下拒绝 select/load plan lifecycle tools，避免 deferred schema 被重新发现。
- `EnterPlanModeTool` / `ExitPlanModeTool` 在 execute 前做 runtime guard，即使过滤层失效也返回错误结果而不修改 approval mode 或 plan gate state。

`CoreToolScheduler` 的 plan-mode blocker 也调整了 subagent/SDK 文案：当 plan mode 阻止非只读工具时，提醒模型直接把计划交回 caller，而不是继续尝试交互式 `exit_plan_mode`。
