## 解决的问题

`qwen serve` 已经通过 fast path 提前 `listen()`，但首个 `/health` 请求仍可能被监听后的 runtime import/evaluation 阻塞。headless YOLO safety warning 也会经 core runtime import 拉入重依赖，导致监控方看到的首个 health 响应接近旧启动时长，而不是 fast path 的监听时长。

## 最终实现方案

`packages/cli/src/serve/run-qwen-serve.ts` 将 runtime 启动拆到 bootstrap health flush 之后：serve 先建立轻量 app、监听端口并响应首个 `/health`，再启动完整 runtime graph。为没有 health probe 的场景增加 fallback timer，保证 runtime 不会无限延迟。

`packages/cli/src/serve/fast-path.ts` 和 `runtime-startup-errors.ts` 保留 fast path 错误分类；`packages/cli/src/utils/headlessSafetyWarnings.ts` 改成轻量判断，避免警告路径提前动态导入 core runtime。测试集中在 `run-qwen-serve.test.ts`、`fast-path.test.ts`、`fast-path-open.test.ts` 与 headless warning 单测，覆盖首个 health 响应、fallback 启动和错误路径。
