## 解决的问题

#6031 落地了 daemon-managed channel worker，但初版更偏启动路径：worker ready 后异常退出、心跳停滞、日志中泄露 bearer token 或代理凭据、超大日志行、pidfile stale worker pid、部分 channel 连接失败等情况，都需要清晰的降级、诊断和恢复策略。

## 最终实现

`packages/cli/src/serve/channel-worker-supervisor.ts` 增加完整运行期 supervisor：ready 前仍 fail-fast；ready 后异常退出会进入有界重启策略，默认 5 分钟窗口最多 3 次，backoff 为 1s、5s、15s。worker 每 15s 通过 IPC 发送 heartbeat，supervisor 45s 未收到 heartbeat 会 kill stale child，记录 `staleHeartbeatAt` 并走同一重启路径。超过策略后升级为 `failed`，`/daemon/status` 暴露 `channel_worker_exited` 等 issue。

日志路径新增 ANSI/control/invisible 字符清理、daemon token 和敏感 env value 脱敏、proxy URL credential 脱敏、单行与 buffer 上限，以及 oversized remainder 的有界丢弃。`packages/channels/base/src/sanitize.ts` 也补了日志文本 sanitizer。snapshot 中的 error 字段统一 sanitizer，避免 status 面泄露敏感值。

pidfile/status 侧在重启窗口内保留 serve ownership 但清空 stale `workerPid`；ready snapshot 带 `requestedChannels` 和实际 connected `channels`，若部分 channel 连接失败但至少一个成功，status issue 标记 `channel_worker_partial_connect`。
