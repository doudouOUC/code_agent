## 解决的问题

ACP `newSession`、`loadSession` 和 `unstable_resumeSession` handler 会并发执行。旧实现把最新加载的 `LoadedSettings` 放在共享字段 `this.settings` 上，并在多段 `await` 后再次读取它来构造 `Config` / `Session` 或检查 runtime output 目录。两个不同 workspace 的请求交错时，先进入的请求可能拿到后进入请求的 settings 实例，导致模型持久化写入错误 workspace，或在错误 `advanced.runtimeOutputDir` 下查找 session。

## 最终实现

`packages/cli/src/acp-integration/acpAgent.ts` 让三个 session 入口在请求开始时各自加载本 workspace 的 `settings`，并把该实例显式传给 `newSessionConfig()` 和 `createAndStoreSession()`。`newSessionConfig()` 不再自己调用 `loadSettings()`，merged snapshot、hook accessor 和 disabled-skill provider 都闭包到请求级 settings；`createAndStoreSession()` 用同一实例构造 `Session`。

`this.settings` 仍保留为 agent 级“最近加载”缓存，供认证持久化、fast model、folder trust、language 和 settings 扩展方法等既有读者使用。`loadSession` / `unstable_resumeSession` 的 session 存在性检查改为使用本请求 settings 调用 `runWithAcpRuntimeOutputDir()`；失败的 404 探测仍不更新全局缓存，保持旧语义。

## 关键代码路径

- `packages/cli/src/acp-integration/acpAgent.ts`: session 入口加载请求级 settings，并显式传递到 config/session 创建。
- `packages/cli/src/acp-integration/acpAgent.test.ts`: 并发 `newSession` 交错回归测试，验证每个 session 拿到自己的 settings 实例。
