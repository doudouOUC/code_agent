## 解决的问题

daemon dashboard 的 preflight auth cell 只检查环境变量。用户通过 `settings.security.auth.apiKey`、provider `envKey` 对应的 `settings.env`，或 CLI flags 提供 API key 时，session 实际可启动，但 preflight 仍提示缺少 `OPENAI_API_KEY` 等环境变量，造成误报。

## 最终实现

`packages/cli/src/acp-integration/acpAgent.ts` 在 preflight auth 检查中保留原有 env-var 优先判断；当 env var 未命中时，fallback 读取 live `Config` 的 `getModelsConfig().getGenerationConfig()?.apiKey`。该值已经汇总环境变量、settings apiKey、provider envKey 和 CLI flag 等凭据来源，与实际 auth/session 启动路径使用的 resolved key 保持一致。

同时修正 `apiKeyVars.length === 0` 的分支：非 env-keyed auth 类型如果已经存在 resolved key，则返回 `ok`；只有没有 resolved key 时才保持 `unknown`。

## 关键代码路径

- `packages/cli/src/acp-integration/acpAgent.ts`: preflight auth status 的 resolved API key fallback。
- `packages/cli/src/acp-integration/acpAgent.test.ts`: 覆盖 settings apiKey、provider envKey 和无 env-key auth type 的 preflight 行为。

## 验证方式

- `cd packages/cli && npx vitest run src/acp-integration/acpAgent.test.ts`
- 手工路径：不设置目标 env var，仅在 settings 中配置 API key 后启动 `qwen serve --open`，确认 dashboard preflight auth cell 显示 `ok` 而不是 warning。
