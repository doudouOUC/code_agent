## 解决的问题

`DaemonChannelBridge.prompt()` 需要在 daemon SSE 事件流中收齐本 turn 的 text chunks 后再返回给 channel adapter。旧实现靠 `setTimeout(0)` 猜测“一个 macrotask tick 足够事件泵处理 late chunks”，这对 ordered SSE stream 来说不够确定，代码里也留下了等待替换为 turn completion 信号的 TODO。

## 最终实现

`packages/channels/base/src/DaemonChannelBridge.ts` 增加 `turnBarriers` map，以及 `createTurnBarrier()`、`resolveTurnBarrier()`、`clearTurnBarrier()`。`prompt()` 发起 session prompt 前为当前 session 建立 barrier，收到 daemon SSE `turn_complete` 时 resolve；收到 `turn_error` 时先通过 `emitProtocolError()` 记录协议错误，再 resolve，避免 prompt 永久等待。

drop、cancel、stop 等会终止 session 或 abort active prompt 的路径也会先 resolve barrier，确保 session 结束时不会留下悬挂 promise。最终代码仍对非 SSE / blocking prompt 路径保留 one-tick fallback：`Promise.race([turnBarrier, setTimeout(0)])`，因此主路径使用确定性 `turn_complete`，兼容路径不被卡死。

## 关键代码路径

- `packages/channels/base/src/DaemonChannelBridge.ts`: turn barrier lifecycle、`turn_complete` / `turn_error` handling、drop/cancel cleanup。
- `packages/channels/base/src/DaemonChannelBridge.test.ts`: deterministic barrier、error barrier、session termination cleanup。

## 验证方式

- `packages/channels/base/src/DaemonChannelBridge.test.ts`: prompt 等待 `turn_complete` 后返回 chunks；`turn_error` 释放等待并记录协议错误；drop/cancel/stop 不留下 pending barrier。
- 回归判断来自 merged diff 与最新 `main`：非 SSE 路径仍保留 fallback，不把实现误写成“完全删除 setTimeout”。
