## 解决的问题

`GET /health?deep=1` 已能通过 bridge getter 反映 daemon 是否有活跃 prompt 和最近活动时间，但更完整的 `GET /daemon/status` 缺少这些 activity 字段。排障页面或 dashboard 消费 `/daemon/status` 时无法判断 daemon 是空闲、正在处理 prompt，还是刚刚有过会话活动。

后续 patch 还发现 full workspace status 中 MCP section 的 summary 只有通用数量，operator 需要无需展开完整 JSON 就能看到 MCP server fleet 中 connected、errored、disabled 的数量。

## 最终实现

`packages/cli/src/serve/daemon-status.ts` 在 `runtime` 下新增 additive `activity` 对象，字段为 `activePrompts`、`lastActivityAt`、`idleSinceMs`。构建 status response 时先读取一次 `bridge.lastActivityAt` 到本地变量，随后用同一值生成 ISO timestamp 与 idle duration，避免同一响应内 timestamp/duration 来自两次不同 getter 读取。`activePrompts` 直接读 `bridge.activePromptCount`。

bootstrap serve app 还没有重 runtime，因此 `run-qwen-serve.ts` 的 bootstrap `/daemon/status` stub 返回 `activePrompts: 0`、`lastActivityAt: null`、`idleSinceMs: null`，保证 runtime shape 在启动前后稳定。

第二个 patch 在 `summarizeStatusData()` 中调用 `summarizeMcpServers()`：如果 workspace section data 包含 `servers` 数组，就统计 `serversConnected`、`serversErrored` 和 `serversDisabled`。disabled 优先于 error/connected 计数；error 看 `status === 'error'`；connected 看 `mcpStatus === 'connected'`。

## 关键代码路径

- `packages/cli/src/serve/daemon-status.ts`: `runtime.activity` status shape、bridge activity getter 读取、MCP server summary。
- `packages/cli/src/serve/run-qwen-serve.ts`: bootstrap `/daemon/status` 的 activity 默认值。
- `docs/developers/qwen-serve-protocol.md`: daemon status protocol 文档补 `runtime.activity` 与 MCP summary 字段。

## 验证方式

- `packages/cli/src/serve/daemon-status.test.ts`: 从未活跃时 activity 为 `0/null/null`；活跃后输出 ISO `lastActivityAt` 和 `idleSinceMs`；bootstrap response 保持相同 shape；MCP summary 统计 connected/error/disabled。
- 手工验证路径：运行 `qwen serve` 后请求 `curl http://127.0.0.1:4170/daemon/status | jq '.runtime.activity'`，发送 prompt 前后观察 active/last/idle 变化。
