## 解决了什么问题

production `serve` entry 已启用 Node module compile cache，但随后由 daemon spawn 的 ACP children 没有继承这个 cache directory。结果 child 仍要重新编译常用模块，前面围绕 ACP static closure 做的懒加载优化无法复用父进程已经建立的编译缓存。

## 最终实现

当前 open diff 在 `scripts/cli-entry.js` 调用 Node compile cache 后，只有当 Node 返回“本进程新启用 cache”时，才把 resolved compile cache directory 发布到 child 环境。用户显式设置、禁用、启用失败、Node 不支持或已有配置时都不覆盖现状，避免意外改变用户选择或污染不兼容 runtime。

`packages/cli/src/cli.test.ts` 新增边界测试，覆盖成功传播、用户已有环境不覆盖、disabled/failed/unsupported 不发布。`docs/design/acp-compile-cache-propagation.md` 记录 startup topology、兼容边界、备选方案和 paired benchmark。

## 关键代码路径

- `scripts/cli-entry.js`: enable compile cache 后发布 cache directory 给 child 环境。
- `packages/cli/src/cli.test.ts`: cache propagation 边界测试。
- `docs/design/acp-compile-cache-propagation.md`: 设计、兼容性和 benchmark 记录。

## 验证方式

- `packages/cli/src/cli.test.ts` 覆盖 compile cache env propagation。
- paired cold-start benchmark 对比 ACP child 能否复用父进程 compile cache。
- build/typecheck/ESLint 按 PR 验证记录执行。
