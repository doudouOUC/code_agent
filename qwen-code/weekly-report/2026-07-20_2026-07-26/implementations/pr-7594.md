## 解决了什么问题

production `serve` entry 已启用 Node module compile cache，但随后由 daemon spawn 的 ACP children 没有继承同一个 cache directory。这样 child 仍要重新编译常用模块，前面围绕 ACP static closure 做的 lazy telemetry、lazy undici、lazy Google GenAI SDK 等优化无法复用父进程已经建立的编译缓存。

## 最终实现

最终实现把 compile cache 启用与环境传播从 `scripts/cli-entry.js` 拆到 `packages/cli/src/config/compile-cache.ts`。父进程只有在 Node runtime 支持 compile cache、没有被用户显式禁用、且能拿到有效 cache directory 时，才把目录写入 child environment；用户已经配置 compile cache、Node 返回 disabled/failed/unsupported 或无法解析目录时，不覆盖现有环境。

传播路径也接入 CLI environment 与 serve fast-path settings。ACP child spawn 继续通过既有 environment construction 继承父级安全 env，只额外携带 compile cache directory；fast-path tests 固定 production serve path 不会漏掉该变量。设计文档记录 startup topology、兼容边界和 paired benchmark，避免后续把该优化误改成总是覆盖用户设置。

## 关键代码路径

- `scripts/cli-entry.js`: production entry 在进程早期启用 compile cache，并调用 helper 发布目录。
- `packages/cli/src/config/compile-cache.ts`: compile cache status 解析、env key 写入、disabled/failed/unsupported 边界。
- `packages/cli/src/config/environment.ts`, `environment.test.ts`: child environment 构造与继承测试。
- `packages/cli/src/serve/fast-path-settings.ts`, `fast-path.test.ts`: serve fast path 下的 child env propagation。
- `packages/cli/src/cli.test.ts`: CLI entry 级 compile cache propagation 边界测试。
- `docs/design/acp-compile-cache-propagation.md`: 启动拓扑、benchmark 与回滚边界。

## 验证方式

- CLI unit tests 覆盖成功传播、用户已有配置不覆盖、disabled/failed/unsupported 不发布。
- environment / fast-path tests 覆盖 ACP child 继承路径。
- paired cold-start benchmark 对比 ACP child 是否复用父进程 compile cache。
- build/typecheck/ESLint 按 PR 验证记录执行。
