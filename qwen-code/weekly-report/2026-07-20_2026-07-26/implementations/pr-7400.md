## 解决的问题

daemon prompt 路由返回 202 之后，SDK/WebUI/IDE 会按 `promptId` 等待 SSE 中的 `turn_complete` 或 `turn_error` 来结算回合。旧实现没有“每个 accepted prompt 恰好一个正式终态”的统一所有权：queued prompt 被 remove 时只发队列 UI 事件，deadline 只在路由侧 abort signal，close/kill/channel crash/shutdown 可能先关 EventBus，最后客户端 detach 又可能落在 active prompt 与 queued prompt 的 FIFO 交接间隙。

这些路径都会让已经返回 202 的 prompt 没有 formal terminal，或在竞态中重复/乱序发布 terminal。结果是 `DaemonClient.prompt()`、`DaemonSessionClient` 的 pending prompt、WebUI turn settlement 或未来 Java daemon SDK 都可能永久等待，也可能静默丢掉已接受的 queued work。

## 最终实现方案

PR 在 bridge 内建立 prompt terminal 的统一发布边界。`PendingPromptEntry` 增加 `terminalPublished` latch，`publishPromptTerminal()` 成为所有正式终态的唯一入口：agent 正常 settle 发布 `turn_complete`，AbortError fallback 发布 `turn_complete{stopReason:'cancelled'}`，错误路径发布 `turn_error`，后续任何竞争路径再次进入都会被 latch 抑制。terminal 的 `originatorClientId` 只取 admission 时记录在 pending entry 上的值，避免调用方在不同路径里给出不一致归因。

`--prompt-deadline-ms` 的所有权从 route timer 迁到 bridge。`POST /session/:id/prompt` 仍解析 `deadlineMs`，并用 `resolvePromptDeadlineMs(serverMs, requestMs)` 计算“服务端上限与请求覆盖的较小值”，但不再设置 route-side abort timer，而是把 `deadlineMs` 传入 `BridgeClientRequestContext`。bridge 在 admission 点 arm deadline，覆盖排队等待和实际执行；到期时发布 `turn_error{code:'prompt_deadline_exceeded'}`，调用 `settleActivePromptState()` 清 active 记账，通过 `deadlinePromise` 释放 per-session FIFO，并 abort pending signal 触发既有 best-effort `connection.cancel()`。

session teardown 路径新增 `flushPromptTerminals(entry, code, message)`。`closeSession()`、`killSession()`、channel crash 和 `shutdown()` 都在发布 `session_closed`/`session_died` 并关闭 EventBus 之前，对 `pendingPromptList` 中 active + queued prompt 逐一发布 error terminal，再 abort 对应 signal。这样 observer 会先看到自己的 prompt terminal，再看到 session lifecycle terminal，后续迟到 settle 由 latch 去重。

queue 与 detach 也改为围绕 pending prompt 计数决策。`removePendingPrompt()` 对 queued prompt 在删除 FIFO 节点时立即发布 cancelled terminal；running prompt 仍走 cooperative cancel 路径。`pendingPromptCount` 取代瞬时的 `promptActive` 作为 close-on-last-detach、attach rollback 和 idle reaper 的门控，覆盖 active 和 queued，也覆盖两个 prompt 交接时 `promptActive` 短暂为 false 的窗口。deferred close-on-prompt-complete 放在 `result.finally()` 中，并要求 terminal broadcast 已经排在它之前、`byId.get(sessionId) === entry` 仍成立、没有 client/subscriber/pending prompt 后才关闭。

## 关键代码路径

- `packages/acp-bridge/src/bridge.ts`: `publishPromptTerminal()`、`flushPromptTerminals()`、deadline race、`pendingPromptCount` close gate、`removePendingPrompt()` cancelled terminal、close/kill/crash/shutdown terminal flush。
- `packages/acp-bridge/src/bridgeErrors.ts`: `PromptDeadlineExceededError` 移入 bridge 包，说明 deadline 由 bridge 拥有。
- `packages/acp-bridge/src/bridgeTypes.ts`: `BridgeClientRequestContext.deadlineMs` 与 `PendingPromptEntry.terminalPublished`。
- `packages/cli/src/serve/routes/session.ts`: prompt route 只解析并转发 effective deadline；不再 route-side arm timer 或 abort signal。
- `packages/cli/src/serve/server/prompt-deadline.ts`: 继续 re-export `PromptDeadlineExceededError`，保持既有 server/test import 兼容。
- `packages/cli/src/serve/server.test.ts`: deadline route 测试改为断言 `context.deadlineMs` 透传，并确认 route 不再 abort bridge signal。
