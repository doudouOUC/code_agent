## 解决了什么问题

daemon / SDK 可靠性审计中的 DAEMON-009/010/011 指向同一个 session event pipeline 风险面：长时间未结束的 turn 会让 compaction live journal 无界增长；EventBus close 对从未开始迭代的 subscriber 清理不完整；不可 JSON 序列化事件和大量 replay backlog 可能绕过 live byte cap，制造内存峰值或诊断缺口。

## 最终实现

EventBus publish 路径现在会在分配 id、写 ring 和推 subscriber 前 eager 计算并 memoize serialized byte length。无法 `JSON.stringify` 的事件会被拒绝发布并输出诊断，不烧 event id，也不污染 ring、subscriber queue 或 compaction journal；同一 byte length 结果复用于 live queue 和 compaction journal，避免不同层各自估算。

replay 增加独立 byte budget，默认是 live subscriber byte cap 的 4 倍。断线客户端重放 retained frames 时，如果 replay burst 超预算，EventBus 会优先保证至少一帧能交付，然后推送 `state_resync_required{reason:'replay_budget_exceeded'}`，订阅保持 live，客户端可按 resync 契约拉 snapshot，而不是让 replay 以 force-push 绕过所有预算。`close()` 统一调用 subscriber `dispose()`，确保未迭代 subscriber 的 AbortSignal listener、queue 与 bus 引用都被释放。

compaction live journal 同步增加 `maxJournalEvents`（默认 2000）与 `maxJournalBytes`（默认 2 MiB）。超过上限时丢弃最老 live journal entries，并插入 `history_truncated{reason:'replay_window_exceeded', scope:'live_journal'}` marker。turn boundary compaction 不受影响，因为它折叠的是 merged working set，而不是无界 live journal。
