## 解决的问题

TypeScript daemon SDK 的 REST SSE 订阅是长生命周期 fetch。旧实现虽然给 parser 传入 signal，并在 parser finally 释放 reader，但 consumer early return、`for await` break、generator throw、stream/connection error 或 transport `dispose()` 时，不一定能 abort 底层 fetch/TCP 连接。部分运行时下 `reader.cancel()` 不足以关闭连接，daemon 侧 EventBus subscriber 可能泄漏；pending iterator read 也可能一直挂住。

## 最终实现方案

`RestSseTransport` 新增 `activeSseRequests: Set<AbortController>`。每次 `subscribeEvents()` 创建 request-lifetime `AbortController` 后立即登记，connect timeout、caller signal 和 request controller 被组合成 fetch signal。连接阶段仍只对“请求到响应头”计时；收到 headers 后取消 connect timer，body 流不套超时。

整个 fetch、HTTP error body 读取、content-type 校验、body parser 和 `yield* parseSseStream()` 都被包在一个 try/finally 中。无论 iterator 自然结束、early return、consumer throw、stream error 还是 connection error，finally 都会 `requestCtrl.abort()` 并从 active set 删除。`dispose()` 幂等设置 closed 状态，并 abort/清空所有 active request；后续新订阅仍按 closed transport 抛错。

`parseSseStream` 的注释和 cleanup 语义也同步调整为 cancel body stream，明确它是 iterator exit 和 AbortSignal cancellation 的资源释放点。实现不替换上游错误：connection refused、stream error、consumer throw 都原样传给调用方，同时确保 request 被 abort。

## 验证方式

- `RestSseTransport.test.ts`: generator `.return()`、`for await` break、`gen.throw()`、stream error、fetch throw 都断言底层 fetch signal aborted。
- 多 SSE request active 时调用 `dispose()`，所有 request signal 均 aborted，并且 `dispose()` 可重复调用。
- dispose 后 pending iterator read 能被释放；仍在 connecting 阶段的 request 会收到 abort。
- 既有 URL encoding、`Last-Event-ID`、`maxQueued`、HTTP error body、content-type 校验和 connect-timeout 测试继续覆盖兼容行为。
