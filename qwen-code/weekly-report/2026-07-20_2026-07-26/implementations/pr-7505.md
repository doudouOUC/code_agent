## 解决的问题

enterprise memory 不能把语义提供商当作事实源，也不能只靠 best-effort tombstone 管删除。需要一个 canonical persistence layer，在 tenant boundary、RLS、事务、删除意图、source receipt 和 anti-resurrection ledger 上先建立不可绕过的事实源。

## 当前实现方案

当前 open diff 在 #7509 foundations 之上新增 `001-initial.sql` migration、`canonical-store.ts` 和 `anti-resurrection-ledger.ts`。PostgreSQL schema 对 tenant-scoped canonical records、raw events、deletion intents、provider receipts、audit/outbox 和 erasure reservation 建表，并强制 RLS。store API 在事务中设置 tenant context，source receipt 更新必须命中当前可见 canonical record；ambiguous commit 会保留外部 key/receipt 供 reconciliation，而删除流程先记录 durable intent 再确认 tombstone。

anti-resurrection ledger 是 content-free 外部删除账本契约，用 irreversible key/receipt 跟踪已删除内容，避免 provider retry、backup restore 或异步 reconciliation 把已删除 memory 重新激活。测试覆盖 RLS migration contract、canonical CRUD/activation/deletion ordering 和 ledger 收敛行为。

## 关键代码路径

- `integrations/enterprise-memory/migrations/001-initial.sql`: 初始 schema、tenant RLS、outbox/audit/erasure reservation。
- `integrations/enterprise-memory/src/canonical-store.ts`, `canonical-store.test.ts`: canonical record persistence、transaction-local tenant context、receipt/deletion flows。
- `integrations/enterprise-memory/src/anti-resurrection-ledger.ts`, `anti-resurrection-ledger.test.ts`: content-free deletion ledger。
- `integrations/enterprise-memory/src/migration-contract.test.ts`: schema/RLS contract。
