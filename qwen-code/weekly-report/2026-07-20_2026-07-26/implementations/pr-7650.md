## 解决了什么问题

部分 OpenAI-compatible streaming provider 会发出这种帧序列：内容 chunk、finish chunk（暂不带 usage）、一个空 `choices` 或无 candidates 的空帧、usage-only chunk、EOF。旧逻辑没有把 no-candidate frame 识别为空帧，导致 pending finish 在 usage 到达前被释放；之后 usage-only frame 已经来不及合并到最终响应，transcript、usage telemetry 和上层消费都会看到缺 usage 的完成事件。

## 最终实现

最终实现调整 OpenAI stream pipeline 的 empty-frame 判断：没有 candidate parts 的 frame 都视为空帧，不论 provider 返回的是空 candidates/choices，还是干脆缺少候选结构。空帧不会触发 pending finish flush，finish-only chunk 会继续缓存到 usage 到达或 EOF；usage-only、tool-call preparation 和其它有效增量仍按既有路径处理。

回归测试覆盖 finish 后插入空帧再到 usage 的 provider 行为，并用 mutation guard 确认“过早 yield finish”不会被后续对象修改掩盖。这样 telemetry 与 transcript 记录的是最终 merged usage，而不是中途 finish snapshot。
