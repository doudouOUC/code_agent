## 解决的问题

现有 Java SDK 主要是 stdio/CLI transport，无法安全处理 daemon HTTP admission 与 SSE 事件之间的竞态、断线续传、事件去重、terminal 关联、权限响应、session teardown 与资源回收。让每个 Java 应用各自实现这些协议细节会重复高风险代码，并可能把部分输出误报为成功。

## 当前实现方案

当前 open diff 在既有 `com.alibaba:qwencode-sdk` Maven artifact 内新增 Java 11 daemon transport，不引入第二个制品。新 API 用 `DaemonClient` 做 HTTP admission/discovery，`DaemonSessionClient` 绑定 thread-scoped session，拆分 prompt admission 与 terminal completion；`promptText()` 提供 bounded convenience API，底层仍保留 typed/raw event observer、permission response、cancel、heartbeat、close/detach/destroy 等控制面。

传输上，prompt 先用非重试 `POST` admission，拿到 watermark 后通过 resumable SSE 读取终态。客户端发送 `Accept-Encoding: identity` 与 `Last-Event-ID`，严格校验 UTF-8/JSON/SSE framing，只在 observer 成功处理后推进 cursor，去重 replayed events，拒绝 event id gap 和 compressed stream。对无法证明成功的 mutation、prompt admission 或 terminal 缺失，使用显式 exception 类型 fail closed。配套 daemon/ACP path 补齐 Java 依赖的可靠性契约：terminal-before-response 缓冲、admission-aware cancel、deadline/queued/running cancel/teardown ordering、permission cancellation 等。
