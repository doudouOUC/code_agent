## 解决的问题

#7276 后 telemetry 初始化变成 async lazy path，daemon metrics 必须等 telemetry init settle 后再初始化，否则 meter provider 尚未注册时记录的 daemon metrics 会绑定到 noop provider。这个顺序原本只靠代码形状保证，没有测试。另一个维护风险是 `NodeSDK` options 里 metrics 只有单数 `metricReader` 字段，未来清理时容易误改成并不存在的 `metricReaders: []` 空数组语义。

## 最终实现方案

`packages/cli/src/serve/run-qwen-serve.test.ts` 新增 ordering test，断言 daemon serve 初始化顺序为 telemetry start → telemetry resolved → daemon metrics initialization。`packages/core/src/telemetry/sdk-impl.ts` 在 NodeSDK options 附近补注释：`@opentelemetry/sdk-node@0.203.0` 只接受单个 `metricReader`，没有 metrics 空数组 opt-out；要压制 env-based reader auto config，只能显式传 reader，这也是 traces/logs processor arrays 与 metrics reader 字段不对称的原因。

## 关键代码路径

- `packages/cli/src/serve/run-qwen-serve.test.ts`: daemon metrics init ordering 回归测试。
- `packages/core/src/telemetry/sdk-impl.ts`: `metricReader` singular option 的 load-bearing 注释。
