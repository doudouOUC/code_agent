## 解决的问题

daemon SSE event id 在进程重启后会从 1 重新开始，旧逻辑只用 `lastEventId >= nextId` 的数字启发式识别 stale cursor。一旦新 epoch 的事件数追上旧游标，客户端可能带着旧 reducer state 静默跳过新 epoch 的前 N 个事件。另一个缺口是 turn-boundary compaction 合并 text/thought/tool-call 后丢失 `promptId`、`originatorClientId` 与 session attribution；compaction engine 出错时 EventBus 为保持 publish 不抛会吞掉异常，但后续 replay snapshot 没有 degraded 信号。

## 当前实现方案

当前 open diff 给每个 EventBus 生成随机 `eventEpoch`，并把它下发到所有客户端获取 cursor 的表面：create/load/resume 响应、非阻塞 prompt 202 envelope、REST SSE 与 `/acp` session stream 的 `X-Qwen-Event-Epoch` header。客户端重连时在 `Last-Event-ID` 旁回传 epoch；若与当前 bus epoch 不一致，daemon 直接走 `state_resync_required{reason:'epoch_reset', detail:'epoch_mismatch'}`，不再依赖数字启发式。

compaction engine 在合并 text/thought chunks 时保留最新的 `promptId`、`originatorClientId` 与 `data.sessionId`，tool-call fold 也改为 latest-wins attribution。EventBus 在 compaction ingest/seed 第一次失败时 latch degraded 状态，snapshot、load response 和 `/acp` initial replay 都透出 degraded 信号，并按 session 写一次 operator breadcrumb。TypeScript daemon SDK 从 response/header 学习 epoch，把它和 cursor 一起保存，并在 `DaemonClient.prompt()` 内部临时 SSE、`DaemonSessionClient` 长连、REST/ACP HTTP transport reconnect 时回传。
