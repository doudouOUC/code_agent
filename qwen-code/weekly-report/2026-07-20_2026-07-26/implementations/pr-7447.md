## 解决的问题

该 PR 与已合入的 #7276 目标一致：默认 telemetry 关闭时，ACP child 仍会静态加载 OpenTelemetry NodeSDK、instrumentation、HTTP/gRPC exporter 和 protobuf；telemetry 开启时也会把两条 OTLP protocol chain 一起拉入启动闭包，增加 daemon cold start 成本。

## 关闭前实现方案

关闭前 diff 已把 `packages/core/src/telemetry/sdk.ts` 拆成轻量 facade 与 heavy `sdk-impl.ts`，并把 OTLP URL helper、HTTP exporter、gRPC exporter 分离到独立模块；`initializeTelemetry()` 走 async single-flight dynamic import，disabled path 不加载 heavy implementation。daemon serve 入口在初始化 daemon metrics 前显式 await telemetry init，普通 Config/startup 入口保留 fire-and-forget。

配套 bundle 侧增加 `scripts/sdk-node-exporter-stub.js`，对 NodeSDK 的 env-var auto exporter eager require 做打桩，并扩展 `scripts/check-serve-fast-path-bundle.js`，防止 telemetry heavy cluster 或另一条 protocol chain 回到 ACP static closure。该 PR 很快关闭，最终落地口径以已合入的 #7276 及其后续 #7456 为准。

## 关键代码路径

- `packages/core/src/telemetry/sdk.ts`, `sdk-impl.ts`: lazy facade 与 heavy NodeSDK assembly。
- `packages/core/src/telemetry/otlp-urls.ts`, `sdk-exporters-http.ts`, `sdk-exporters-grpc.ts`: protocol-specific exporter split。
- `packages/cli/src/serve/run-qwen-serve.ts`, `packages/cli/src/startup/startup-prefetch.ts`: async telemetry init call sites。
- `esbuild.config.js`, `scripts/sdk-node-exporter-stub.js`, `scripts/check-serve-fast-path-bundle.js`: bundle stub 与 static closure guard。
