## 解决的问题

telemetry 懒加载之后，`undici` 仍是 ACP child static startup closure 里的大块第三方成本。代理 dispatcher、API preconnect、IDE client fetch、GitHub setup、自更新、web search 等路径只有真正发起网络请求时才需要 undici，但旧代码在启动模块求值期就静态导入，导致默认 cold start 付出约 2MiB eager closure 成本。

## 最终实现方案

CLI 与 core 各自新增 package-local `load-undici` helper，用 single-flight dynamic import 延迟加载 undici，并兼容 esbuild 打包后 CommonJS default-only chunk 形态。两个包刻意不共享 helper，因为它们解析各自的 undici 副本，测试 mock 也需要各自命中。

所有静态导入点改为运行时加载：代理 dispatcher 安装、channel proxy、API preconnect、GitHub setup、自更新、IDE client、web search 与 runtime fetch options。`Config` 初始化和 channel start 仍 await 代理 dispatcher 安装，保证相对首个网络请求的顺序不变。bundle guard 新增 undici blacklist，任何静态 undici import 回到 ACP eager closure 都会让检查失败。

## 关键代码路径

- `packages/cli/src/utils/load-undici.ts`, `packages/core/src/utils/runtimeFetchOptions.ts`: package-local dynamic import 与 proxy dispatcher 组装。
- `packages/cli/src/utils/apiPreconnect.ts`, `packages/cli/src/commands/channel/start.ts`, `packages/cli/src/commands/channel/proxy.ts`: preconnect/channel proxy 改为 lazy undici。
- `packages/core/src/ide/ide-client.ts`, `packages/core/src/tools/web-search.ts`, `packages/cli/src/services/setup-github.ts`, `standalone-update.ts`: 网络调用导入点延迟化。
- `scripts/check-serve-fast-path-bundle.js`: ACP static closure 中 undici 静态导入守卫。
- `docs/design/2026-07-21-lazy-undici-loading.md`: 设计与 benchmark 记录。
