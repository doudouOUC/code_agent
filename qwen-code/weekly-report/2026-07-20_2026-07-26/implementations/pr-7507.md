## 解决的问题

持久化层只定义数据事实源，不能回答谁可以共享、评审、召回或删除 memory，也不能定义语义 provider 写入、发现、删除、孤儿对账如何收敛。需要在 canonical store 上实现受治理的 memory lifecycle，并保证 provider output 永远不会成为身份、授权或 canonical truth。

## 当前实现方案

当前 open diff 新增 `memory-service.ts`、`mem0-semantic-index.ts` 与 `privacy-mode-store.ts`。Memory service 覆盖 capture、candidate proposal、consent/review、activation、authorized recall、tombstone、erasure、provider outbox reconciliation 和 privacy preferences。候选激活先在 canonical store 中预留 operation，再写 provider，且只有 exact provider discovery 成功后才变为 recallable。

Recall 先走 provider-neutral semantic index contract，再把 provider result 映射回 active canonical record，并重新执行 tenant、scope、ACL、tombstone、expiry、deletion intent、lease/current authorization 检查；无法建立 canonical authorization、deletion-ledger state 或 provider reconciliation 时 fail closed。Mem0 adapter 被严格当作可替换索引，测试使用 deterministic doubles 覆盖 retry、并发 consent 变化、lease expiry、删除竞态、malformed pagination 和 orphan reconciliation。

## 关键代码路径

- `integrations/enterprise-memory/src/memory-service.ts`, `memory-service.test.ts`: governed lifecycle、activation/recall/delete/reconciliation。
- `integrations/enterprise-memory/src/mem0-semantic-index.ts`, `mem0-semantic-index.test.ts`: provider-neutral semantic index implementation and Mem0 adapter contract。
- `integrations/enterprise-memory/src/privacy-mode-store.ts`: privacy preference storage。
