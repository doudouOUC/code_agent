## 解决的问题

`enter_plan_mode` 会在模型 turn 的中途切换 approval mode。旧实现只给模型返回一段短句，完整 Plan mode 约束要等后续系统提示才更可靠；如果同一个模型响应里还包含其它工具调用，这些 sibling tool call 可能在模式切换前后继续执行，没有经过新的 Plan boundary 重新评估。

这会让模型一边进入 Plan mode，一边继续执行同批写操作或其它需要重新确认的工具。对 ACP 和 headless 来说，这个问题更明显：客户端需要为模型响应中的每个 tool call 得到一个终态响应，但又不能让 sibling 在边界之外偷偷执行。

## 最终实现方案

PR 新增 `packages/core/src/core/plan-mode-entry-policy.ts`，导出 `findPlanModeEntryBatchBoundaryIndex()` 和统一的 sibling skip 文案。Core scheduler 在 call-id dedupe 与 canonical tool name 解析之后、权限检查和工具构造之前定位第一个 `enter_plan_mode`。只允许该调用执行；同批其它 executable call 直接生成 `EXECUTION_DENIED` tool response，要求模型在看到新的 approval mode 后下一轮重试。

ACP session 的批处理路径复用同一 policy。它先完成 loop/duplicate provider call 处理，再把 first `enter_plan_mode` 所属批次作为唯一可执行批次；其它批次里的调用都会记录 skipped tool result 并发出 tool-call lifecycle 事件。headless 非交互路径也在 duplicate/structured-output 过滤后应用同样边界，skipped call 不消耗 `--max-tool-calls` 执行预算。

`enter_plan_mode` 本身的成功和幂等返回改为完整 `getPlanModeSystemReminder()`，并根据 SDK mode 选择对应文案。为保证这段 lifecycle policy 不被普通输出治理破坏，该工具声明 `maxOutputChars = Infinity`，Core scheduler 的 persistence spill gate 和 aggregate offload 也把 `ENTER_PLAN_MODE` 作为语义豁免。失败、YOLO unsolicited entry、subagent/team 拒绝仍保留原有失败结果，因为这些路径没有成功进入 Plan mode。

## 关键代码路径

- `packages/core/src/core/plan-mode-entry-policy.ts`: Plan entry boundary index 和统一 sibling skip message。
- `packages/core/src/core/coreToolScheduler.ts`: dedupe 后执行边界、sibling `EXECUTION_DENIED`、`enter_plan_mode` 输出 spill/offload 豁免。
- `packages/core/src/tools/enterPlanMode.ts`: 成功/幂等返回完整 Plan mode system reminder，并声明无限 per-tool output limit。
- `packages/cli/src/acp-integration/session/Session.ts`: ACP batches 中按同一 boundary 跳过 sibling，并保持 duplicate responses 有序。
- `packages/cli/src/nonInteractiveCli.ts`: headless batch 在 duplicate/structured-output 处理后应用 Plan entry boundary。
