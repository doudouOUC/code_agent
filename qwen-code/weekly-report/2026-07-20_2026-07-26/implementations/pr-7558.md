## 解决了什么问题

#7276 已把 telemetry SDK 变成 lazy facade，但 ACP child 在 protocol initialize 阶段仍会在写出 initialize response 前启动 telemetry 初始化。对 daemon channel cold startup 来说，host 只需要尽快收到 initialize result；telemetry import/SDK init 应该排在这个协议响应之后，否则会把 OpenTelemetry 成本留在用户可见的 channel startup path。

## 最终实现

最终实现把 ACP telemetry init 移到成功写出匹配 request id 的 initialize result 之后。`runAcpAgent` 在 NDJSON transport 边界记录 incoming `initialize` request id，只有当 response 是同一 request id、发送成功且 result 成功时，才触发既有 single-flight telemetry facade。这样 telemetry 仍在 ACP child 内启动，但不阻塞 initialize response。

`loadCliConfig` 的 ACP startup 模式改为不在 config load 时 eager initialize telemetry；普通 interactive TUI、prompt-interactive、headless 和 daemon parent 路径保留原有时序。测试锁定成功 initialize 后启动、非匹配 request/失败 response 不启动、shutdown cleanup 不被破坏，以及 config 层 ACP defer 标志。
