## 解决的问题

daemon `single` session 允许多个客户端 attach 到同一个活跃会话。旧实现用 `attachCount` 表示 spawn 之后又 attach 进来的客户端数，但 `detachClient()` 不校验 detach 是否真的对应一份 attach 引用：重复 detach、未知 clientId、匿名 detach，甚至 spawn owner 用自己的 clientId detach，都会尝试递减 `attachCount`。

这会把别的真实 attacher 的计数偷走。典型后果是 spawn-owner 断连时留下的 `spawnOwnerWantedKill` tombstone 或 close-on-last-detach 路径观察到 `attachCount === 0`，提前 kill 仍有 live client 的 session，后续请求全部变成 404。

## 最终实现方案

`SessionEntry` 增加 `attachRefs: Map<string, number>`，把“哪个 clientId 拥有几份 attach-after-spawn 引用”作为 `attachCount` 的账本。只有对 `attachCount` 有真实贡献的路径才调用 `recordAttachRef()`：命中 `defaultEntry` 的普通 attach、spawn/restore coalesce 后的 attached waiter、restore raced existing entry 等。owner 型注册不进入账本，因为 spawn owner 和 restore 发起者本来不贡献 `attachCount`。

`detachClient(sessionId, clientId)` 改为先尝试 `releaseAttachRef(entry, clientId)`。只有账本确实释放了一份引用时才允许 `attachCount--`；重复 detach、未知 clientId、匿名请求和 owner clientId 都不会影响 `attachCount`。随后仍无条件调用 `unregisterClient()`，保持 client registration 的幂等清理语义，并让 owner 明确离开后 close-on-last-detach 仍可到达。

失败回滚也对齐同一纪律。`rollbackAttachRegistration()` 先释放发起者 clientId 的账本引用；如果存在 coalesce reservation 这种尚未注册 clientId 的预留，则用 `attachCountDelta - 1` 扣除这部分未登记贡献。这样 restore 发起者、coalesce waiter 和普通 attach 的失败清理都只扣自己贡献，不会影响其他客户端。
