## 解决的问题

ACP child cold start 的静态 import closure 里仍包含 OpenTelemetry NodeSDK、instrumentation、gRPC、protobuf、OTLP transformer 和各类 exporter。默认 telemetry 关闭时，这些模块仍会被 parse/compile 后再由 `initializeTelemetry()` 短路。telemetry 开启时，HTTP 和 gRPC exporter 链也会同时进入同一个实现 closure，即使配置只会使用其中一种 protocol。

这会把遥测的模块加载成本放到 daemon cold first-session 和 `channel.initialize` 关键路径上，尤其在 2C4G 机器上影响明显。

## 最终实现方案

最终实现把 `packages/core/src/telemetry/sdk.ts` 拆成轻量 facade 和 heavy implementation。facade 只保留 cheap runtime state：`sdk`、`telemetryInitialized`、`telemetryInitPromise`、`telemetryShutdownPromise`、`activeMetricReader`、`isTelemetrySdkInitialized()`、`refreshSessionContext()`、`shutdownTelemetry()`、`forceFlushMetrics()` 和 diag logger。它的 runtime OTel import 只剩 `@opentelemetry/api`。

`initializeTelemetry(config)` 改为 async single-flight。它先检查 `telemetryInitialized || !config.getTelemetryEnabled()`，disabled path 直接 resolved，不触发 dynamic import；开启时才 `await import('./sdk-impl.js')` 并调用 `startTelemetrySdk(config)`。Config constructor 和 startup prefetch 走 fire-and-forget 并记录 rejection；`run-qwen-serve` 在 `initializeDaemonMetrics()` 前显式 `await core.initializeTelemetry(...)`，避免 global MeterProvider 尚未注册时 daemon metrics 被永久绑定到 noop meter。

heavy `sdk-impl.ts` 接收原先 NodeSDK、resource、propagator、instrumentation、feedback-loop guard、session processor、file/log-to-span 和 exporter 装配。OTLP URL 解析迁到 `otlp-urls.ts`；HTTP exporter 链和 gRPC exporter 链分别进入 `sdk-exporters-http.ts`、`sdk-exporters-grpc.ts`，由 `sdk-impl.ts` 按 `telemetry.otlpProtocol` 动态 import。`outfile` 路径不加载 OTLP；misconfigured gRPC without base endpoint 跳过且不加载 protocol chain。

bundle 侧新增 `scripts/sdk-node-exporter-stub.js` 和 esbuild plugin，stub 掉 NodeSDK 通过 env-var auto exporter config eager require 的 exporter packages。stub 只应在 unsupported env-var auto config 被真正构造时抛错，防止 silent localhost export。`scripts/check-serve-fast-path-bundle.js` 新增 telemetry blacklist 和 SDK implementation protocol boundary 检查，避免 heavy telemetry 或另一条 protocol chain 回到 ACP static closure。

## 关键代码路径

- `packages/core/src/telemetry/sdk.ts`: facade state、async single-flight `initializeTelemetry()`、pending init aware shutdown/flush。
- `packages/core/src/telemetry/sdk-impl.ts`: heavy NodeSDK assembly、resource/session processor、instrumentation、propagator、signal routing。
- `packages/core/src/telemetry/otlp-urls.ts`: HTTP OTLP signal URL resolution。
- `packages/core/src/telemetry/sdk-exporters-http.ts`, `sdk-exporters-grpc.ts`: protocol-specific exporter factory。
- `packages/core/src/config/config.ts`, `packages/cli/src/startup/startup-prefetch.ts`, `packages/cli/src/serve/run-qwen-serve.ts`: async init call-site behavior。
- `esbuild.config.js`, `scripts/sdk-node-exporter-stub.js`, `scripts/check-serve-fast-path-bundle.js`: bundle stubbing and closure guard。

## 验证方式

- `packages/core/src/telemetry/sdk.test.ts`: disabled path 不加载 heavy implementation、single-flight init、shutdown pending init、HTTP/gRPC/outfile route semantics。
- `packages/core/src/telemetry/telemetry.test.ts`: public telemetry exports 兼容。
- `scripts/tests/sdk-node-exporter-stub.test.js`: stub regex、own-module import 豁免、Windows/POSIX importer 兼容。
